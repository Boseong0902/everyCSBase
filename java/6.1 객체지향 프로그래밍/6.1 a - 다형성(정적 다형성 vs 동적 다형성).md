다형성은 **같은 타입이나 같은 메서드 호출이 상황에 따라 다른 동작으로 이어지는 성질**이다.

자바는 다형성을 자동으로 보장하기보다 다형성을 구현할 수 있는 문법과 런타임 기능을 제공한다. 실제 다형성은 이러한 수단을 어떻게 조합하여 설계하는지에 따라 달라진다.

## 정적 다형성과 동적 다형성

두 다형성의 핵심적인 차이는 **어떤 메서드를 호출할지 결정하는 시점**이다.

### 정적 다형성

컴파일 시점에 호출할 메서드가 결정된다. 컴파일러는 참조 변수와 인자의 **정적 타입**을 기준으로 호출 대상을 선택한다. 대표적인 예가 오버로딩이다.

### 동적 다형성

런타임에 호출할 메서드가 결정된다. 참조 변수의 정적 타입이 아니라 그 변수가 참조하는 **실제 객체의 타입**을 기준으로 오버라이딩된 메서드를 선택한다. 이를 동적 디스패치라고 한다.

| 구분 | 정적 다형성 | 동적 다형성 |
| --- | --- | --- |
| 결정 시점 | 컴파일 시점 | 런타임 |
| 결정 기준 | 변수와 인자의 정적 타입 | 실제 객체의 타입 |
| 대표 방법 | 오버로딩 | 오버라이딩 |
| 디스패치 | 정적 디스패치 | 동적 디스패치 |

## 오버로딩과 오버라이딩

### 오버로딩: 정적 다형성

같은 이름의 메서드를 매개변수의 타입이나 개수를 달리하여 선언한다.

```java
void print(int value) {}
void print(String value) {}
```

호출 대상은 런타임의 실제 객체가 아니라 컴파일 시점에 확인할 수 있는 인자의 정적 타입으로 결정된다.

```java
void print(Object value) {}
void print(String value) {}

Object value = "hello";
print(value); // print(Object) 호출
```

`value`가 참조하는 실제 객체는 `String`이지만, 정적 타입이 `Object`이므로 `print(Object)`가 선택된다.

### 오버라이딩: 동적 다형성

하위 클래스가 부모 클래스나 인터페이스의 메서드를 같은 시그니처로 재정의한다.

```java
class Animal {
    void sound() {
        System.out.println("동물 소리");
    }
}

class Dog extends Animal {
    @Override
    void sound() {
        System.out.println("멍멍");
    }
}

Animal animal = new Dog();
animal.sound(); // 멍멍
```

컴파일 시점에는 `Animal`에 `sound()`가 선언되어 있는지 확인한다. 런타임에는 실제 객체의 타입인 `Dog`를 기준으로 `Dog.sound()`를 실행한다.

| 구분 | 오버로딩 | 오버라이딩 |
| --- | --- | --- |
| 메서드 정의 | 이름은 같고 매개변수는 다름 | 이름과 매개변수가 같음 |
| 필요한 관계 | 같은 클래스 또는 상속 관계 | 상속 또는 인터페이스 구현 관계 |
| 선택 기준 | 인자의 정적 타입 | 실제 객체의 타입 |
| 선택 시점 | 컴파일 시점 | 런타임 |
| 다형성 분류 | 정적 다형성 | 동적 다형성 |

- 즉, Dog에 어떤 객체가 오는지에 따라 sound()의 동작이 달라진다. 
- 그리고 이게 런타임 시점에 정해진다

## 동적 다형성을 구현하고 활용하는 방법
[[6.1 b - 추상 클래스 vs 인터페이스]]

### 1. 인터페이스 구현과 오버라이딩

인터페이스는 서로 다른 구현체를 하나의 공통 타입으로 다룰 수 있게 한다.

```java
interface PaymentService {
    void pay(int amount);
}

class CardPaymentService implements PaymentService {
    @Override
    public void pay(int amount) {
        System.out.println("카드 결제: " + amount);
    }
}

class AccountPaymentService implements PaymentService {
    @Override
    public void pay(int amount) {
        System.out.println("계좌이체: " + amount);
    }
}
```

```java
PaymentService service = new CardPaymentService();
service.pay(10_000);
```

컴파일러는 `PaymentService`에 `pay()`가 선언되어 있는지 확인한다. 실제로 실행할 메서드는 런타임의 실제 객체인 `CardPaymentService`를 기준으로 결정된다. 이를 **동적 디스패치**라고 한다.

### 2. 클래스 상속과 오버라이딩

부모 클래스 타입으로 여러 하위 클래스의 객체를 다룰 수 있다.

```java
class Animal {
    void sound() {
        System.out.println("동물 소리");
    }
}

class Dog extends Animal {
    @Override
    void sound() {
        System.out.println("멍멍");
    }
}

class Cat extends Animal {
    @Override
    void sound() {
        System.out.println("야옹");
    }
}
```

```java
Animal animal = new Dog();
animal.sound(); // 멍멍
```

참조 변수의 정적 타입은 `Animal`이지만, 런타임의 실제 타입이 `Dog`이므로 `Dog.sound()`가 호출된다.

상속은 다형성을 제공하지만 부모와 자식을 강하게 결합한다. 공통 구현을 재사용해야 하는 경우가 아니라면 클래스 상속보다 인터페이스와 조합을 우선적으로 고려한다.

### 3. 의존성 주입

오버라이딩만 사용한다고 호출자와 구현체가 자동으로 느슨하게 결합되는 것은 아니다.

```java
class OrderService {
    private final PaymentService paymentService;

    OrderService(PaymentService paymentService) {
        this.paymentService = paymentService;
    }

    void order(int amount) {
        paymentService.pay(amount);
    }
}
```

```java
OrderService cardOrderService =
        new OrderService(new CardPaymentService());

OrderService accountOrderService =
        new OrderService(new AccountPaymentService());
```

`OrderService`는 구체 클래스가 아닌 `PaymentService`에 의존한다. 생성자를 통해 실제 구현체를 외부에서 주입하므로 `OrderService`를 수정하지 않고도 결제 방식을 교체할 수 있다.

따라서 의존성 주입은 새로운 다형성을 만드는 문법이라기보다, **오버라이딩으로 만든 런타임 다형성을 실제 설계에서 활용하는 방법**이다.

## 매개변수 다형성: 제네릭

제네릭은 타입을 매개변수로 받아 하나의 코드를 여러 타입에 적용하게 한다. 이를 **매개변수 다형성**이라고 한다.

```java
static <T> T first(List<T> values) {
    return values.get(0);
}

String firstName = first(List.of("Kim", "Lee"));
Integer firstNumber = first(List.of(10, 20));
```

하나의 `first()` 알고리즘이 `String`, `Integer` 등 여러 타입에 대해 동작한다. 다만 이는 실제 객체에 따라 재정의된 메서드가 선택되는 런타임 다형성과는 다른 개념이다.

## 정리

| 방법 | 분류 | 결정 시점 | 역할 |
| --- | --- | --- | --- |
| 인터페이스 구현 + 오버라이딩 | 서브타입 다형성 | 런타임 | 서로 다른 구현체를 공통 타입으로 다룬 |
| 클래스 상속 + 오버라이딩 | 서브타입 다형성 | 런타임 | 실제 하위 객체의 메서드를 실행 |
| 의존성 주입 | 런타임 다형성의 활용 | 구성 시점 | 호출자와 구현체를 분리 |
| 오버로딩 | 정적 다형성 | 컴파일 시점 | 인자에 따라 다른 메서드를 선택 |
| 제네릭 | 매개변수 다형성 | 컴파일 시점에 타입 검사 | 하나의 로직을 여러 타입에 재사용 |

런타임 다형성의 핵심은 다음과 같다.

```text
추상 타입으로 참조
→ 여러 구현체가 같은 메서드를 오버라이딩
→ 런타임에 실제 객체의 메서드를 선택
```

여기에 구현체를 외부에서 주입하는 설계를 적용하면 다형성을 활용하면서 구체 클래스에 대한 결합도도 낮출 수 있다.
