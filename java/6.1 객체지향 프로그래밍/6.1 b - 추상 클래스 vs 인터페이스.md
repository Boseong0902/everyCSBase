# 추상 클래스 vs 인터페이스

추상 클래스와 인터페이스는 모두 완성되지 않은 메서드를 선언하고, 하위 타입을 공통 타입으로 다룰 수 있게 한다. 그러나 두 기능의 설계 목적은 다르다.

- **추상 클래스**: 강한 관련이 있는 하위 클래스들이 공통 상태와 구현을 공유하는 기반 클래스
- **인터페이스**: 서로 다른 클래스들이 공통으로 수행할 수 있는 역할이나 기능의 계약

핵심 판단 기준은 다음과 같다.

> 공통 상태와 구현을 공유하는 하나의 클래스 계층을 만들려면 추상 클래스를, 여러 종류의 객체에 공통 역할을 부여하려면 인터페이스를 사용한다.

## 문법과 기능 비교

| 구분 | 추상 클래스 | 인터페이스 |
| --- | --- | --- |
| 주요 목적 | 공통 상태와 구현 재사용 | 역할과 행동의 계약 정의 |
| 관계 | `is-a` 기반의 강한 계층 관계 | `can-do` 역할 관계로도 사용 가능 |
| 인스턴스 필드 | 가능 | 불가능 |
| 생성자 | 사용 가능 | 사용 불가능 |
| 메서드 | 추상 메서드와 구현 메서드 | 추상, `default`, `static`, `private` 메서드 |
| 접근 제어자 | 다양하게 사용 가능 | 추상 메서드는 실질적으로 `public` |
| 상수 | 모든 종류의 필드 선언 가능 | 필드는 `public static final` 상수만 가능 |
| 상속/구현 개수 | 클래스 하나만 상속 가능 | 여러 인터페이스 구현 가능 |
| 적합한 변경 | 공통 상태와 템플릿 로직의 변경 | 역할의 구현체 추가와 교체 |

## 추상 클래스

추상 클래스는 직접 인스턴스화할 수 없는 클래스이다. 추상 메서드뿐 아니라 상태, 생성자, 완성된 메서드를 가질 수 있다.

```java
abstract class Account {
    private int balance;

    protected Account(int balance) {
        if (balance < 0) {
            throw new IllegalArgumentException();
        }
        this.balance = balance;
    }

    public final void withdraw(int amount) {
        validate(amount);
        balance -= amount;
        afterWithdraw(amount);
    }

    private void validate(int amount) {
        if (amount <= 0 || amount > balance) {
            throw new IllegalArgumentException();
        }
    }

    protected abstract void afterWithdraw(int amount);
}

class SavingsAccount extends Account {
    SavingsAccount(int balance) {
        super(balance);
    }

    @Override
    protected void afterWithdraw(int amount) {
        System.out.println("적금 계좌 출금: " + amount);
    }
}
```

`Account`는 계좌의 공통 상태와 출금 절차를 직접 관리한다. 하위 클래스는 공통 절차를 재사용하고 필요한 부분만 구현한다.

### 장점

- 인스턴스 필드로 공통 상태를 보관할 수 있다.
- 생성자로 공통 상태의 초기화와 불변 조건을 강제할 수 있다.
- `protected` 멤버로 하위 클래스에만 필요한 기능을 공유할 수 있다.
- 템플릿 메서드 패턴처럼 공통 알고리즘은 고정하고 일부 단계만 하위 클래스에 맡길 수 있다.
- 공통 코드를 하나의 기반 클래스에서 재사용할 수 있다.

### 단점

- 자바는 클래스 다중 상속을 허용하지 않으므로 상속 기회를 하나 소모한다.
- 부모의 상태와 구현에 의존하므로 하위 클래스와의 결합도가 높다.
- 부모 클래스의 변경이 모든 하위 클래스에 영향을 줄 수 있다.
- 서로 관계없는 클래스에 같은 역할만 부여하기에는 부적합하다.

### 이럴 때 사용한다

- 하위 클래스들이 명확한 `is-a` 관계이다.
- 여러 하위 클래스가 공통으로 유지해야 하는 상태가 있다.
- 공통 로직의 재사용이 핵심 목적이다.
- 하위 클래스의 초기화 과정을 통제해야 한다.
- 공통 알고리즘의 전체 흐름은 고정하고 특정 단계만 하위 클래스가 정의해야 한다.

## 인터페이스

인터페이스는 객체가 제공해야 하는 기능의 계약을 정의한다. 구현 클래스의 계층과 무관하게 공통 역할을 부여할 수 있다.

```java
interface Flyable {
    void fly();
}

class Bird implements Flyable {
    @Override
    public void fly() {
        System.out.println("날개로 난다");
    }
}

class Drone implements Flyable {
    @Override
    public void fly() {
        System.out.println("프로펄러로 난다");
    }
}
```

`Bird`와 `Drone`은 공통 부모나 상태를 가지지 않지만, 모두 날 수 있다는 `Flyable` 역할을 제공한다.

```java
void startFlight(Flyable flyable) {
    flyable.fly();
}
```

호출자는 `Bird`나 `Drone`과 같은 구체 클래스가 아닌 `Flyable` 계약에 의존한다.

### 장점

- 하나의 클래스가 여러 인터페이스를 구현하여 다양한 역할을 가질 수 있다.
- 상속 계층이 다른 클래스들에도 같은 기능 계약을 적용할 수 있다.
- 호출자가 구체 클래스 대신 추상 타입에 의존하게 하여 구현체를 교체하기 쉽다.
- 새로운 구현체를 추가하기 쉽고 테스트 대역을 만들기도 쉽다.
- 구현보다 역할과 행동을 중심으로 API를 설계하게 한다.

### 단점

- 인스턴스 상태를 가질 수 없으므로 공통 상태를 공유하려면 별도의 클래스나 조합이 필요하다.
- 인터페이스에 새로운 추상 메서드를 추가하면 모든 구현체가 영향을 받는다.
- `default` 메서드로 기본 구현을 제공할 수 있지만, 인스턴스 상태를 사용하는 공통 로직을 담기에는 제약이 있다.
- 여러 인터페이스가 같은 시그니처의 `default` 메서드를 제공하면 구현 클래스가 충돌을 직접 해결해야 한다.
- 지나치게 큰 인터페이스는 구현체에 필요 없는 메서드까지 강제할 수 있다.

### 이럴 때 사용한다

- 서로 다른 클래스에 공통 역할을 부여해야 한다.
- 하나의 클래스가 여러 역할을 수행해야 한다.
- 호출자와 구현체 사이에 안정적인 계약을 만들고 싶다.
- 런타임에 구현체를 교체하거나 의존성 주입을 사용해야 한다.
- 프레임워크나 라이브러리가 구현해야 할 확장 지점을 제공해야 한다.

## 선택 기준

### 추상 클래스를 선택한다

다음 질문 중 여러 개에 `예`라고 답할 수 있다면 추상 클래스가 적합하다.

1. 하위 클래스들이 같은 개념의 특수한 종류인가?
2. 하위 클래스들이 공통 상태를 가져야 하는가?
3. 생성자를 통해 공통된 초기화 규칙을 강제해야 하는가?
4. 공통 알고리즘의 상당 부분을 재사용해야 하는가?
5. 하위 클래스에만 노출할 `protected` 기능이 필요한가?

### 인터페이스를 선택한다

다음 질문 중 여러 개에 `예`라고 답할 수 있다면 인터페이스가 적합하다.

1. 공통 상태보다 공통 행동이 중요한가?
2. 상속 계층이 다른 클래스들에 같은 역할을 부여해야 하는가?
3. 하나의 클래스가 여러 역할을 가져야 하는가?
4. 구현체를 자유롭게 추가하거나 교체해야 하는가?
5. 테스트 대역이나 외부 확장 지점이 필요한가?

## 둘을 함께 사용하는 경우

추상 클래스와 인터페이스는 서로 배타적인 선택지가 아니다. 추상 클래스가 공통 구현을 제공하면서 인터페이스를 구현할 수 있다.

```java
interface PaymentService {
    void pay(int amount);
}

abstract class ValidatingPaymentService implements PaymentService {
    @Override
    public final void pay(int amount) {
        if (amount <= 0) {
            throw new IllegalArgumentException();
        }
        executePayment(amount);
    }

    protected abstract void executePayment(int amount);
}

class CardPaymentService extends ValidatingPaymentService {
    @Override
    protected void executePayment(int amount) {
        System.out.println("카드 결제: " + amount);
    }
}
```

- 호출자는 `PaymentService` 인터페이스에 의존한다.
- `ValidatingPaymentService`는 공통된 검증과 실행 절차를 재사용한다.
- 각 구현체는 결제 방식만 구현한다.

이러한 구조는 **외부에는 인터페이스를 계약으로 제공하고, 내부에서는 추상 클래스로 구현을 재사용**하고 싶을 때 적합하다.

## 실전 선택 원칙

1. 단순히 공통 메서드가 필요하다는 이유만으로 상속하지 않는다.
2. 호출자가 알아야 할 계약은 인터페이스로 표현하는 것을 우선적으로 고려한다.
3. 상태, 초기화, 공통 알고리즘을 강하게 통제해야 할 때 추상 클래스를 선택한다.
4. 코드 재사용만이 목적이라면 상속보다 조합을 먼저 검토한다.
5. 초기에 인터페이스를 너무 큰 단위로 설계하지 말고, 클라이언트가 필요로 하는 작은 역할 단위로 분리한다.

한 문장으로 정리하면 다음과 같다.

> **인터페이스는 객체가 무엇을 할 수 있는지를 정의하고, 추상 클래스는 관련된 객체들이 무엇을 공유하는지를 정의한다.**
