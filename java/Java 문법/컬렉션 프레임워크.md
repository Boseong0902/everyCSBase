# 컬렉션 프레임워크

컬렉션 프레임워크는 여러 데이터를 다루기 위한 Java 표준 자료구조 API다.

## 주요 분류

### List

중복을 허용하고 순서가 있다.

### Set

중복을 허용하지 않는다. 순서 보장 여부는 구현체마다 다르다.

- `HashSet`: 순서를 보장하지 않는다.
- `LinkedHashSet`: 삽입 순서를 유지한다.
- `TreeSet`: 값 기준으로 정렬한다.

### Queue

FIFO 또는 우선순위 기반 처리를 위해 사용한다.

### Map

`Map`은 `Collection`을 직접 구현하지는 않는다. key-value 쌍을 저장한다.

- `HashMap`: 해시 기반이며 순서를 보장하지 않는다.
- `LinkedHashMap`: 삽입 순서를 유지한다.
- `TreeMap`: key 기준으로 정렬한다.

## 동시성 컬렉션

멀티 스레드 환경에서 안전하게 사용할 수 있는 컬렉션이다.

`ConcurrentHashMap`은 락을 분할해서 같은 키에 대한 동시 접근을 제한하고, 전체 맵에 대한 락 경합을 줄인다.

## Set과 Map의 유니크 판단

`Set`과 `Map`은 `hashCode()`와 `equals()`를 함께 사용해서 유니크 여부를 판단한다.

- `hashCode()`: 먼저 객체의 해시값으로 버킷을 찾는다.
- `equals()`: 같은 버킷 안에 값이 있을 때 실제 동등성을 확인한다.

기본 `hashCode()`와 `equals()`는 값 기반 비교가 아닐 수 있으므로, 값 객체를 컬렉션 키나 Set 원소로 사용할 때는 둘을 함께 오버라이딩해야 한다.

자세한 비교 기준과 오버라이딩 예시는 [[equals와 ==]]에 정리되어 있다.

## TreeSet과 TreeMap의 정렬 기준

정렬 기준은 `Comparable`의 `compareTo()`를 구현하거나, 외부에서 `Comparator`의 `compare()`를 주입해서 정한다.

둘이 함께 있으면 보통 `Comparator`가 우선된다.

## ArrayList와 LinkedList

`ArrayList`는 연속된 메모리 구조를 사용한다.

`LinkedList`는 각 노드가 이전 노드와 다음 노드의 참조를 저장한다.

| 작업 | ArrayList | LinkedList |
| --- | --- | --- |
| 인덱스 조회 | O(1) | O(n) |
| 맨 앞 추가 | O(n) | O(1) |
| 맨 뒤 추가 | O(1) | O(1) |
| 값 위치 찾기 | O(n) | O(n) |

캐시 지역성까지 고려하면 많은 경우 `ArrayList`가 유리하다.

## HashMap 내부 동작

`HashMap`은 key-value 쌍을 노드로 저장하고, 해시값을 기준으로 버킷에 배치한다.

여러 노드가 같은 버킷에 배정되면 버킷 충돌이 발생한다. 충돌이 많아지면 조회 시간이 O(n)에 가까워질 수 있다.

Java 8 이후에는 한 버킷의 노드 수가 일정 기준을 넘으면 리스트를 트리로 바꿔 조회 시간을 O(log n)으로 줄인다.

## 실무 사용 예시

제품 가격 조회에서 제품 1회 조회 후 원재료별 가격을 매번 조회하면 N+1 쿼리가 생길 수 있다.

먼저 제품의 `ingredient_id` 목록을 가져오고, 두 번째 쿼리에서 `IN` 절로 한 번에 가격을 조회한 뒤 `Map<재료ID, 가격>` 형태로 매핑하면 조회 비용을 줄일 수 있다.

```sql
SELECT *
FROM ingredient
WHERE id IN (1, 2, 3, 4, 5);
```

List로 찾으면 O(n)이지만, Map으로 찾으면 O(1)에 가깝게 조회할 수 있다.
