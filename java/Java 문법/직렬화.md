# 직렬화

직렬화는 객체를 바이트 형태로 변환하는 것이다.

JVM은 참조 타입 데이터에 대해 실제 값 자체가 아니라 그 값이 저장된 위치를 가리키는 참조값을 저장한다. 객체의 멤버 변수도 다른 객체를 참조할 수 있기 때문에, 그대로 외부로 보내면 의미 있는 데이터가 아니라 현재 JVM 안에서만 유효한 참조 구조를 보내게 된다.

그래서 서버 외부로 데이터를 보내거나 저장하려면, 의미 있는 값의 형태로 변환해야 한다.

## 사용 상황

- 사용자 세션을 캐시에 저장
- 블루 그린 배포 중 상태 전달
- MSA 환경에서 서비스 간 데이터 전달
- 메시지 큐를 통한 객체 데이터 전달

## 방법

### Java 네이티브 직렬화

Java 서버 간 전달에는 사용할 수 있다.

다만 `readObject()`를 통해 역직렬화할 때 악성 코드 실행 위험이 생길 수 있다. 외부에서 들어온 데이터를 바로 역직렬화하지 말고, `ObjectInputFilter`로 허용 클래스 화이트리스트를 지정하는 식의 방어가 필요하다.

### Jackson JSON 직렬화

JSON은 언어와 무관하게 통용되기 쉽다.

Java 네이티브 직렬화보다 비교적 가볍고, 데이터 포맷이므로 코드 실행 위험이 낮다. 실무에서는 보통 JSON으로 파싱해서 전달하는 방식을 많이 사용한다.

Java 환경에서도 API 통신이나 서비스 간 데이터 교환에서는 JSON을 사실상의 표준 직렬화 포맷처럼 사용하는 경우가 많다.

- 받는 쪽의 개발 환경과 무관하게 통신하기 쉽다.
- 사람이 읽을 수 있는 텍스트 포맷이라 디버깅이 쉽다.
- Java 네이티브 직렬화보다 언어 종속성이 낮다.
- 실행 코드가 아니라 단순 데이터 구조를 표현하는 포맷이라, 네이티브 역직렬화보다 코드 실행 위험이 낮다.

다만 JSON도 역직렬화 설정을 잘못하면 위험할 수 있다. 예를 들어 타입 정보를 외부 입력으로 받아 다형적 역직렬화를 허용하면, 의도하지 않은 타입이 생성되는 문제가 생길 수 있다.

## Java 네이티브 역직렬화가 위험한 이유


| 자바 네이티브 역직렬화 과정
	역직렬화 -> Java 네이티브 직렬화 데이터에 “새로운 악성 코드”가 그대로 들어 있고 JVM이 그 코드를 실행하는 것은 아니다.
	- 직렬화 데이터에 기록된 클래스 정보를 바탕으로 클래스를 복원한다
	- 이 때, 이미 존재하는 클래스를 다시 객체로 복원하게 될 수도 있다
	- 여기서 해당 클래스의 메서드등을 통해서 서버 내부 자원까지도 접근이 가능해진다(가젯 체인)

공격자는 이 흐름을 악용해 서버에 이미 존재하는 클래스들의 조합, 즉 gadget chain을 유도할 수 있다. 그래서 서버가 신뢰할 수 없는 데이터를 읽는 것만으로도 의도하지 않은 코드 경로가 실행될 수 있다.

따라서 외부 입력을 Java 네이티브 역직렬화로 바로 읽는 것은 피하고, 불가피하다면 `ObjectInputFilter` 같은 필터로 허용할 클래스와 데이터 크기 등을 제한해야 한다.
