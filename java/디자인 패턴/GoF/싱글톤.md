# 싱글톤

인스턴스를 하나만 만든다

어디서든 그 하나의 인스턴스에 접근할 수 있게 한다

커넥션 풀, 스프링 빈, 설정 관리자 등 공유 자원을 중앙 제어해야할 때 사용 -> 객체가 여러 개 있을 필요가 없거나, 여러 개 있으면 문제가 되는 객체에 사용

```java
public class Singleton {
    private static final Singleton INSTANCE = new Singleton();

    private Singleton() {}

    public static Singleton getInstance() {
        return INSTANCE;
    }
}
```

`private static final Singleton INSTANCE = new Singleton();`

-> 클래스 내부에서 자기 자신의 인스턴스를 하나 만든다

`private Singleton(){}`

-> 생성자를 private으로 막아서 외부에서 객체 생성을 못하게 막는다

`public static Singleton getInstance()`

-> 외부에서는 getInstance() 로만 같은 객체를 가져간다

## 목적

싱글톤의 목적은 이런 설정 사항들을 공유하는 등의 이유로 여러 객체가 만들어지면 동시성 문제가 터질 수 있는 관리자 객체들에 한해서 전역에서 하나의 객체만 만들어놓고 호출 함수만 public 으로 열어놓는 디자인 패턴

## 대표 구현 방식

### 미리 생성해두기

미리 생성해둔 만큼, 사용되지 않는 경우 낭비가 될 수 있음

### 필요할때 생성하기

if문으로 해당 객체가 존재하는지 여부 확인 후 생성

멀티스레드 환경에서 동시성 문제가 있음 -> 객체가 여러개 생성될 수 있음

### 필요할때 생성하되, synchronized 키워드 사용

매번 getInstance()를 호출할 때 동기화 비용이 발생

이미 인스턴스가 생성된 이후에도 계속 락을 검사함

### 해당 객체가 존재하지 않을 때에만 synchronized 사용하여 동기화하여 객체 생성

```java
public class Singleton {
    private static volatile Singleton instance;

    private Singleton() {
    }

    public static Singleton getInsatnce() {
        if (instance == null) {
            synchronized (Singleton.class) {
                if (instance == null) {
                    instance = new Singleton();
                }
            }
        }

        return instance;
    }
}
```

이중 if문(두 차례 모두 객체 존재 여부 검사하는 동일한 과정임)

- 첫 번째 - 이미 객체가 있으면 락을 걸지 않고 바로 반환
- 두 번째 - 동시에 여러 스레드가 들어왔을 때, 락 안에서 다시 한번 확인해서 중복 생성을 막음

코드가 복잡

### static inner class

```java
public class Singleton {
    private Singleton() {
    }

    private static class Holder {
        private static final Singleton INSTANCE = new Singleton();
    }

    public static Singleton getInstance() {
        return Holder.INSTANCE;
    }
}
```

### Enum Singleton

```java
public enum Singleton {
    INSTANCE;

    public void doSomething() {
        System.out.println("작업 수행");
    }
}
```

```java
Singleton.INSTANCE.doSomething();
```

## 싱글톤 패턴의 장단점

전역 상태로 부터 발생할 수 있는 문제

어디서든 같은 객체를 바꿀 수 있으며, 어느 코드가 상태를 변경했는지 추적하기 어려움

클래스 내부에서 싱글톤 객체를 직접 호출하면, 테스트 시, 가짜 객체로 바꾸기 어려움

강한 결합으로 이루어져있다

싱글톤 관련 구체클래스의 명칭과 그 안에서의 비지니스 로직을 호출부에서 다 알고있다

인스턴스가 한개뿐이기 때문에 강한결합일수밖에 없다

## 스프링에서의 싱글톤

`@bean` 객체는 컨테이너당 한개로 관리

클래스 생성 자체는 싱글톤을 고려하지 않아도 됨

스프링 자체에서 알아서 싱글톤처럼 객체를 하나로만 유지해서 관리

생성자 주입으로 객체를 주입해줌

어떤 클래스가 다른 객체에 의존한다면, 그 의존 객체를 클래스 내부에서 직접 new 하지 않고, 생성자 파라미터로 받아서 사용함

-> 즉, 생성자 파라미터로 필요한 객체를 언급해놓기만 하면 스프링 컨테이너가 알아서 주입해줌

일반적으로 이렇게 생성자 주입 방식에선 해당 생성자 메서드(파라미터로 객체 주입받는 생성자)에 @Autowired 어노테이션을 붙임

하지만, 정의해놓은 생성자가 하나뿐이면 생략 가능
