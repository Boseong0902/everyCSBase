# 이벤트 리스너와 트랜잭션

키워드 - `@EventListener`, `@TransactionalEventListener`

이 페이지는 [[옵저버]] 패턴을 스프링에서 구현할 때, 후속 작업(알림 구독자)과 트랜잭션의 원자성을 어떻게 맞추는지에 대한 내용이다.

## 문제 상황

예를 들어 주문 완료 이벤트를 만들어보자

```java
@Service
public class OrderService {
    private final ApplicationEventPublisher publisher;

    @Transactional
    public void order(Long orderId) {
        orderRepository.save(new Order(orderId));

        publisher.publishEvent(new OrderCompletedEvent(orderId));
    }
}

// 그리고 EventListener도 만들어보자
@Component
public class EmailListener {
    @EventListener
    public void handle(OrderCompletedEvent event) {
        emailService.send(event.getOrderId());
    }
}
```

이 리스너는 이벤트가 발행되는 순간 실행된다

즉, 실행 흐름이 다음과 같아진다

트랜잭션 시작 -> 주문 저장 -> 이벤트 발행 -> EmailListener 실행 -> 이메일 발송 -> 트랜잭션 커밋

근데 만약, 이메일이 발송되고 나서 트랜잭션이 문제가 생겨서 롤백된다면?

이게 이메일 발송 뿐만 아니라, 이벤트 리스너가 만약 재고 차감, 결제 요청 등등이었다면 문제가 더 커지겠지

-> DB에는 주문 기록이 없는데, 해당 주문에 대한 이메일이 발송되고, 재고가 차감되고, 결제가 요청되게 된다

- 이런 문제를 줄이기 위해 `@TransactionalEventListener`를 사용한다

```java
@Component
public class EmailListener {
    @TransactionalEventListener(phase = TransactionPhase.AFTER_COMMIT)
    public void handle(OrderCompletedEvent event) {
        emailService.send(event.getOrderId());
    }
}
```

## @TransactionalEventListener

Spring의 이벤트 리스너를 트랜잭션 상태와 연결해서 실행시키는 어노테이션

- 일반 `@EventListener`는 이벤트가 발행되면 바로 실행된다
- 앞에서도 봤지만, 이는 트랜잭션 롤백 가능성 때문에 문제가 된다
- 따라서 스프링은 트랜잭션과 연동되는 이벤트 리스너, 즉 TransactionalEventListener를 제공

### phase

어느 시점에 TransactionalEventListener가 실행될지를 정하는 옵션

```java
@TransactionalEventListener(phase = TransactionPhase.AFTER_COMMIT)
public void handle(OrderCompletedEvent event) {
    ...
}
```

이런 식으로 설정

### phase 종류

- BEFORE_COMMIT (커밋 전 실행)
	- 트랜잭션 시작 -> 이벤트 발행 -> 커밋 직전 리스너 실행 -> 커밋
	- 사용 예시
		- 커밋 전에 추가 DB변경
		- 커밋 전에 마지막 검증
		- 같은 트랜잭션 안에서 커밋 전에 해야될 로직적인 무언가
	- 여기서 예외가 발생하면 커밋이 실패하고 롤백됨
- AFTER_COMMIT (커밋 후 실행)
	- 트랜잭션 시작 -> 이벤트 발행 -> 커밋 성공 -> 리스너 실행
	- 사용 예시
		- 알림 발송
		- 외부 API 호출
	- DB커밋이 성공한 뒤 실행하는게 안전한 작업들
- AFTER_ROLLBACK (롤백 후 실행)
	- 트랜잭션 시작 -> 이벤트 발행 -> 예외 발생 -> 롤백 -> 리스너 실행
	- 사용 예시
		- 실패 로그 기록
		- 실패 알림
		- 임시 리소스 정리
		- 보상 트랜잭션
- AFTER_COMPLETION (트랜잭션이 끝나면 무조건 실행)
	- 트랜잭션 작업이 끝난 뒤 실행 -> 커밋 성공 후, 혹은 롤백 후 실행
	- 사용 예시
		- 공통 정리 작업
		- 리소스 해제
		- 감사 로그
		- 처리 완료 기록

## 주의점

이건 사실상 이벤트 발행과 트랜잭션을 분리하는 작업이기 때문에, 후속 작업이 실패해도 트랜잭션이 롤백되진 않는다.

-> 따라서 이에 대해선 재시도 로직을 고려하는것이 중요하다(재시도, 실패로그, 메시지 큐, outbox패턴 등 고려 가능)

이렇게 분리를 한다 해도, 기본적으론 동기 처리 흐름이다. 따라서 비동기를 원하면 추가적으로 `@Async`를 붙여줘야 함

- 하지만, 비동기로 만들면, 후속 작업의 예외 발생 시, 이 예외가 호출자에게 직접 전파되지 않으므로 로그 혹은 재시도 로직이 필요해진다 -> outbox패턴, 메시지 큐 등등

이렇게 하나의 로컬 트랜잭션 경계를 넘어 여러 작업의 정합성을 맞추는 문제는 결국 [[분산 트랜잭션]]으로 이어진다.
