# 정적 팩토리 메서드와 팩토리 메서드

객체 생성을 사용하는 쪽에서 직접 하지 않고, 생성 책임을 다른 곳으로 분리하는 패턴

즉, new를 어디서 하냐에 대한 차이를 만든다

## no Factory Pattern

```java
class OrderService {
    public void pay(String type) {
        Payment payment;

        if (type.equals("CARD")) {
            payment = new CardPayment();
        } else if (type.equals("KAKAO")) {
            payment = new KakaoPayment();
        } else {
            throw new IllegalArgumentException();
        }

        payment.pay();
    }
}
```

이 코드는 OrderService가 주문 처리도 하고, 결제 객체를 만들기도 함 -> SRP위반

이 코드는 다른 결제 수단, 즉 다른 클래스가 추가되어야 할 땐, 이 호출부를 직접 수정해야한다 -> OCP위반

## Simple Factory

```java
class PaymentFactory {
    public Payment create(String type) {
        if (type.equals("CARD")) {
            return new CardPayment();
        }

        if (type.equals("KAKAO")) {
            return new KakaoPayment();
        }

        throw new IllegalArgumentException();
    }
}

class OrderService {
    private final PaymentFactory paymentFactory;

    public OrderService(PaymentFactory paymentFactory) {
        this.paymentFactory = paymentFactory;
    }

    public void pay(String type) {
        Payment payment = paymentFactory.create(type);
        payment.pay();
    }
}
```

이게 기본 아이디어이다 -> 객체를 사용하는 쪽은 생성 과정을 몰라도 되고 그냥 이거에 맞는 객체 만들어달라고 요청하면 된다 -> 추상화

이건 GoF의 팩토리 패턴은 아니다만, 기본 개념은 여기서 출발하기에 언급해봤다

## Factory Method Pattern

위에서 simple Factory pattern을 확인했다

기본 개념은 저기서 나오지만, 저 패턴의 단점은 명확하다 -> 새 타입이 추가될 때마다 PaymentFactory의 조건문을 수정해야된다. 즉, 완벽하게 OCP를 보장하진 못한다

그렇다면 Factory Method pattern이란?

객체들이 종류와 상관없이 공통된 작업이 여러개 있다면,

- 그 작업들은 부모 클래스가 처리,
- 객체 생성은 자식 클래스에게 맡기는 패턴

```java
abstract class PaymentProcessor {
    public void process() {
        Payment payment = createPayment();
        payment.pay();
        payment.paidCheck();
        payment.refund();
    }

    protected abstract Payment createPayment();
}

class CardPaymentProcessor extends PaymentProcessor {
    @Override
    protected Payment createPayment() {
        return new CardPayment();
    }
}

class KakaoPaymentProcessor extends PaymentProcessor {
    @Override
    protected Payment createPayment() {
        return new KakaoPayment();
    }
}
```

즉, 공통 작업은 부모가 정의하고, 객체 생성만 자식에게 위임한다

위 코드에선

- PaymentProcessor에서 공통 작업인 pay, paidCheck, refund를 구현
- 어떤 객체가 주입될건지는 자식 클래스에 위임
- 자식 클래스가 만들어서 주입하는 객체에 따라서 pay, paidCheck, refund가 어떤 동작을 할지는 Payment 추상클래스와 그 구체 클래스에서 구현

즉, 어떤 객체든 공통 작업이 여러개 있다면, 이 작업(메서드)들을 하나의 추상 클래스에 묶어돔

## Abstract Factory Pattern

관련 있는 객체 여러 개를 한 세트로 생성하는 패턴

예:

- window UI와 mac UI를 구분해서 만들어야되는 상황
- 두 OS UI 모두 button과 checkbox객체가 필요

```java
interface Button {
    void render();
}

interface Checkbox {
    void render();
}

interface UIFactory {
    Button createButton();
    Checkbox createChecbox();
}
```

이제 이 관련 객체들을 일괄적으로 만드는 공장을 정의

클라이언트가 구체 클래스에 의존하지 않은 채 팩토리를 사용하여 연관된 객체들을 일괄적으로 생성할 수 있도록 하는 패턴

## 총정리

### 팩토리 패턴

객체 생성 책임을 객체를 사용하는 코드에서 분리하고자 하는 생성 패턴

### 팩토리 메서드 패턴

객체 종류와는 별도로 공통된 작업들을 일괄적으로 처리하고 싶을때,

해당 공통 작업들은 부모가 처리, 객체 생성은 자식이 처리

### 추상 팩토리 패턴

연관된 객체들이 있는경우, 즉 일괄적으로 생성하는 객체들이 있는 경우 이를 하나의 생성팩토리로 묶고, 클라이언트가 구체 클래스에 의존하지 않은 채 팩토리를 사용하여 연관된 객체들을 일괄적으로 생성할 수 있도록 하는 패턴
