# 정적 팩토리 메서드와 팩토리 메서드

객체 생성 책임을 객체를 사용하는 코드에서 분리하는 방법들은 이름이 비슷해서 헷갈리기 쉽다.

먼저 용어를 분리해야 한다.

| 이름 | GoF 패턴 여부 | 핵심 |
| --- | --- | --- |
| 정적 팩토리 메서드 | GoF 패턴 아님 | `static` 메서드로 객체 생성 의도를 드러내고 생성 과정을 숨긴다 |
| 단순 팩토리 | GoF 패턴 아님 | 별도 Factory 객체가 조건에 따라 구현체를 생성한다 |
| 팩토리 메서드 패턴 | GoF 생성 패턴 | 객체 생성 메서드를 부모에 두고, 실제 생성은 자식 클래스가 결정한다 |
| 추상 팩토리 패턴 | GoF 생성 패턴 | 관련 객체들을 하나의 제품군으로 묶어 생성한다 |

공통 관심사는 `new`를 어디서 할 것인가이다. 하지만 각각이 해결하는 문제와 구조는 다르다.

## 팩토리 없이 직접 생성하는 코드

```java
class OrderService {
    public void pay(String type) {
        Payment payment;

        if (type.equals("CARD")) {
            payment = new CardPayment();
        } else if (type.equals("KAKAO")) {
            payment = new KakaoPayment();
        } else {
            throw new IllegalArgumentException();
        }

        payment.pay();
    }
}
```

이 코드의 문제는 `OrderService`가 주문 처리와 결제 객체 생성을 동시에 맡는다는 점이다.

- 주문 비지니스 로직과 객체 생성 로직이 섞인다.
- 결제 수단이 추가될 때 `OrderService`를 수정해야 한다.
- [[SRP - 단일 책임]]과 [[OCP - 개방-폐쇄]]를 지키기 어렵다.

## 정적 팩토리 메서드

정적 팩토리 메서드는 생성자 대신 `static` 메서드로 객체를 생성하게 하는 방식이다.

GoF의 팩토리 메서드 패턴과는 다른 개념이다.

```java
class Money {
    private final int amount;

    private Money(int amount) {
        this.amount = amount;
    }

    public static Money won(int amount) {
        return new Money(amount);
    }
}
```

```java
Money price = Money.won(10_000);
```

생성자 `new Money(10000)`보다 `Money.won(10000)`이 더 의미를 잘 드러낸다.

정적 팩토리 메서드의 장점은 다음과 같다.

- 생성자보다 이름으로 생성 의도를 표현할 수 있다.
- 생성자를 `private`으로 막고 생성 경로를 통제할 수 있다.
- 항상 새 객체를 만들지 않고 캐싱된 객체나 기존 객체를 반환할 수 있다.
- 반환 타입을 인터페이스나 부모 타입으로 숨길 수 있다.

```java
interface Payment {
    void pay();
}

class PaymentFactory {
    public static Payment card() {
        return new CardPayment();
    }

    public static Payment kakao() {
        return new KakaoPayment();
    }
}
```

여기서 `PaymentFactory.card()`는 정적 팩토리 메서드다. 하지만 이것을 GoF의 팩토리 메서드 패턴이라고 부르면 부정확하다.

## 단순 팩토리

단순 팩토리는 객체 생성 책임을 별도 Factory 클래스로 옮기는 방식이다.

GoF 패턴은 아니지만, 팩토리 개념을 이해하기 위한 출발점으로 자주 사용된다.

```java
class PaymentFactory {
    public Payment create(String type) {
        if (type.equals("CARD")) {
            return new CardPayment();
        }

        if (type.equals("KAKAO")) {
            return new KakaoPayment();
        }

        throw new IllegalArgumentException();
    }
}

class OrderService {
    private final PaymentFactory paymentFactory;

    public OrderService(PaymentFactory paymentFactory) {
        this.paymentFactory = paymentFactory;
    }

    public void pay(String type) {
        Payment payment = paymentFactory.create(type);
        payment.pay();
    }
}
```

이제 `OrderService`는 `new CardPayment()`를 직접 하지 않는다. 객체 생성 책임은 `PaymentFactory`로 이동했다.

하지만 결제 수단이 추가될 때마다 `PaymentFactory`의 조건문을 수정해야 한다. 그래서 단순 팩토리는 생성 책임 분리에는 도움이 되지만, [[OCP - 개방-폐쇄]]를 완전히 만족시키지는 못한다.

## 팩토리 메서드 패턴

팩토리 메서드 패턴은 GoF 생성 패턴이다.

공통 처리 흐름은 부모 클래스가 갖고, 어떤 객체를 만들지는 자식 클래스가 결정한다.

```java
abstract class PaymentProcessor {
    public void process() {
        Payment payment = createPayment();
        payment.pay();
        payment.paidCheck();
        payment.refund();
    }

    protected abstract Payment createPayment();
}

class CardPaymentProcessor extends PaymentProcessor {
    @Override
    protected Payment createPayment() {
        return new CardPayment();
    }
}

class KakaoPaymentProcessor extends PaymentProcessor {
    @Override
    protected Payment createPayment() {
        return new KakaoPayment();
    }
}
```

핵심은 `process()`의 흐름은 부모가 고정하고, `createPayment()`만 자식이 바꾼다는 점이다.

- `PaymentProcessor`: 공통 알고리즘을 가진 Creator
- `createPayment()`: 객체 생성을 자식에게 위임하는 팩토리 메서드
- `CardPaymentProcessor`, `KakaoPaymentProcessor`: 구체 객체 생성을 결정하는 ConcreteCreator

새 결제 방식이 필요하면 새로운 `PaymentProcessor` 하위 클래스를 추가한다. 기존 부모의 공통 처리 흐름은 유지된다.

## 추상 팩토리 패턴

추상 팩토리 패턴도 GoF 생성 패턴이다.

팩토리 메서드 패턴이 “객체 하나의 생성 책임을 자식에게 위임”하는 데 초점이 있다면, 추상 팩토리 패턴은 “관련 객체 여러 개를 하나의 제품군으로 생성”하는 데 초점이 있다.

예를 들어 Windows UI와 Mac UI를 구분해서 만들어야 하고, 각 UI 제품군마다 Button과 Checkbox가 필요하다고 하자.

```java
interface Button {
    void render();
}

interface Checkbox {
    void render();
}

interface UIFactory {
    Button createButton();
    Checkbox createCheckbox();
}

class WindowsButton implements Button{
	@Override
	public void render(){
		System.out.println("Windows 버튼 렌더링");
	}
}

...
```

```java
class WindowsUIFactory implements UIFactory {
    public Button createButton() {
        return new WindowsButton();
    }

    public Checkbox createCheckbox() {
        return new WindowsCheckbox();
    }
}

class MacUIFactory implements UIFactory {
    public Button createButton() {
        return new MacButton();
    }

    public Checkbox createCheckbox() {
        return new MacCheckbox();
    }
}
```

클라이언트는 `UIFactory`만 알면 된다. 구체 제품이 Windows 계열인지 Mac 계열인지는 팩토리 구현체가 결정한다.

## 비교 정리

| 구분 | 생성 책임 위치 | 주로 해결하는 문제 |
| --- | --- | --- |
| 정적 팩토리 메서드 | 클래스의 `static` 메서드 | 생성 의도 표현, 생성자 통제, 반환 타입 숨김 |
| 단순 팩토리 | 별도 Factory 객체 | 사용하는 코드에서 `new` 제거 |
| 팩토리 메서드 패턴 | 자식 Creator 클래스 | 공통 흐름은 유지하고 생성만 하위 클래스에 위임 |
| 추상 팩토리 패턴 | 제품군 Factory 구현체 | 관련 객체들을 일관된 제품군으로 생성 |

## Spring에서는 어떻게 연결되는가

팩토리 계열 패턴의 공통 목적은 객체를 사용하는 코드에서 `new`를 직접 하지 않게 만들고, 생성 책임을 다른 곳으로 분리하는 것이다.

Spring에서는 이 역할을 [[DI와 IoC|스프링 컨테이너]]가 많이 대신한다.

```java
interface Payment {
    void pay();
}

@Component
class CardPayment implements Payment {
    @Override
    public void pay() {
        System.out.println("카드 결제");
    }
}

@Component
class KakaoPayment implements Payment {
    @Override
    public void pay() {
        System.out.println("카카오 결제");
    }
}

@Service
class OrderService {
    private final Payment payment;

    OrderService(Payment payment) {
        this.payment = payment;
    }

    void order() {
        payment.pay();
    }
}
```

`OrderService`는 `new CardPayment()`나 `new KakaoPayment()`를 직접 호출하지 않는다. 객체 생성과 주입은 스프링 컨테이너가 담당한다.

즉, `OrderService`는 구체 클래스가 아니라 `Payment`라는 추상화에 의존한다. 이 점에서 [[DIP - 의존 관계 역전]]을 만족시키기 쉬워진다.

구현체가 여러 개이면 스프링은 어떤 Bean을 주입해야 할지 결정할 수 없으므로 `@Primary`, `@Qualifier`, Bean 이름, 설정 클래스 등을 통해 선택 기준을 알려줘야 한다.

```java
@Service
class OrderService {
    private final Payment payment;

    OrderService(@Qualifier("kakaoPayment") Payment payment) {
        this.payment = payment;
    }
}
```

## @Bean 메서드와 정적 팩토리 메서드

정적 팩토리 메서드처럼 생성 과정을 메서드 안에 숨기고 싶다면 `@Configuration`과 `@Bean`을 사용할 수 있다.

```java
@Configuration
class PaymentConfig {
    @Bean
    Payment payment() {
        return new KakaoPayment();
    }
}
```

클라이언트 코드는 여전히 `Payment`만 주입받는다. 실제로 어떤 구현체가 만들어지는지는 설정 클래스가 결정한다.

## 여러 전략을 선택해야 할 때

결제 타입에 따라 구현체를 선택해야 한다면, Spring은 같은 타입의 Bean들을 `Map`으로 주입할 수 있다.

```java
@Service
class PaymentService {
    private final Map<String, Payment> payments;

    PaymentService(Map<String, Payment> payments) {
        this.payments = payments;
    }

    void pay(String type) {
        Payment payment = payments.get(type);
        payment.pay();
    }
}
```

이 방식은 `if (type.equals(...))` 조건문을 직접 늘리는 단순 팩토리보다 확장에 유리하다. 새로운 결제 구현체를 Bean으로 추가하면, 사용하는 쪽은 `Payment` 인터페이스와 Bean 이름만 기준으로 선택할 수 있다.
