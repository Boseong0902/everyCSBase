# 어댑터

호환되지 않는 인터페이스를 클라이언트가 기대하는 인터페이스로 변환해주는 구조 패턴

-> 이미 존재하는 클래스가 있는데(외부 호출 등의 이유로), 내가 원하는 메서드 이름이나 사용 방식과 맞지 않을 때 그 사이에 어댑터를 둔다

## 구성 요소

- Target - client가 기대하는 인터페이스
- Adaptee - 기존 클래스(외부 라이브러리, 외부 의존성 등)
- Adapter - Target을 구현하고 내부에서 Adaptee를 호출하는 클래스(기존 클래스, 외부 클래스 등을 우리 로직에 맞추기 위해 끼우는 클래스)
- Client - Target만 바라보는 호출부(우리 애플리케이션의 비지니스 로직)

## 예시

결제 기능을 개발하는데, 카드 결제, 카카오 결제, 네이버페이 결제, 토스페이 결제 기능을 넣으려고 한다

근데 각각의 결제 요청을 보내는 메서드가 달라서 우리 비지니스 로직, 즉 호출부에서 외부에 강하게 의존해야되는 상황이 발생한다

-> 이런 경우, 다형성의 문제가 발생하게 된다.

- 주입되는 객체에 따라 오버라이드된 메서드의 동작이 달라지게 구현해야되는데,
- 이런 경우, 어떤 객체를 주입받는지에 따라 다른 메서드를 호출할 수 있도록 if-else문으로 구현해야된다

즉, OCP원칙을 위배하게 된다

```java
if (type.equals("KAKAO")) {
    kakaoClient.requestPay(amount);
} else if (type.equals("NAVER")) {
    naverClient.approvePayment(amount);
} else if (type.equals("TOSS")) {
    tossClient.executeBilling(amount);
}
// 이거보단,
// ------------------------------------------
paymentService.pay(amount);
// 이렇게 구현해야되니까
```

따라서 다음과 같은 어댑터 클래스를 추가해준다

```java
public interface PaymentService {
    void pay(int amount);
}

public class KakaoPayAdapter implements PaymentService {
    private final KakaoClient kakaoClient;

    @Override
    public void pay(int amount) {
        kakaoClient.requestPay(amount);
    }
}

public class NaverPayAdapter implements PaymentService {
    private final NaverClient naverClient;

    @Override
    public void pay(int amount) {
        naverClient.approvePayment(amount);
    }
}

public class TossPayAdapter implements PaymentService {
    private final TossClient tossClient;

    @Override
    public void pay(int amount) {
        tossClient.executeBilling(amount);
    }
}
```

## 정리

어댑터는 단순히 메서드명만 바꾸는 것이 아니라, 파라미터, 반환값, 예외 처리 방식, 외부 용어를 우리 도메인에 맞게 변환하는 역할도 한다
