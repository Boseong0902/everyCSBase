# 전략

상황에 따라 바뀔 수 있는 알고리즘이나 정책을 별도의 객체로 분리하여 실행 시점에 갈아끼울 수 있게 만드는 행동 패턴

## 예시

할인 정책을 구현해본다고 하자

```java
public class OrderService {
    public int calculateDiscount(String type, int price) {
        if (type.equals("FIXED")) {
            return 1000;
        }
        if (type.equals("RATE")) {
            return price / 10;
        }
        if (type.equals("VIP")) {
            return price / 20 + 2000;
        }

        return 0;
    }
}
```

이는 동작은 하지만,

- 할인 정책이 추가될 때마다 OrderService 수정 -> OCP위반
- 조건문 증가 -> 안티패턴
- 테스트 복잡도 증가

따라서 이를 인터페이스 & 구체 클래스 형식으로 분리

```java
public interface DiscountStrategy {
    int discount(int price);
}

public class FixedDiscountStrategy implements DiscountStrategy {
    @Override
    public int discount(int price) {
        return price / 10;
    }
}
...

public class OrderService {
    private final DiscountStrategy discountStrategy;

    public OrderService(DiscountStrategy discountStrategy) {
        this.discountStrategy = discountStrategy;
    }

    public int order(int price) {
        int discountAmount = discountStrategy.discount(price);
        return price - discountAmount;
    }
}
```

전략패턴은 대표적인 다형성을 보장하기 위한 패턴

-> 전략이 추가될 때마다 호출부 코드는 수정을 지양하고, 전략 인터페이스를 구현하는 구체클래스를 확장하는 형식으로 기능 확장

## 언제 사용?

- 같은 목적의 알고리즘이 여러개이다
- 조건문으로 알고리즘을 선택하고있다
- 알고리즘이 자주 추가되거나 변경된다
- 실행 시점에 정책을 바꿔야 한다
- 알고리즘을 사용하는 코드와 구현 로직을 분리하고 싶다

## Java환경에서는

요청에 따라 전략을 선택하는 상황이라면, Map으로 사용도 가능하다

```java
@Service
public class PaymentService {
    private final Map<String, PaymentStrategy> strategies;

    public PaymentService(Map<String, PaymentStrategy> strategies) {
        this.strategies = strategies;
    }

    public void pay(String type, int amount) {
        PaymentStrategy strategy = strategies.get(type);

        if (strategy == null) {
            throw new IllegalArgumentException("지원하지 않는 결제 방식");
        }

        strategy.pay(amount);
    }
}
```
