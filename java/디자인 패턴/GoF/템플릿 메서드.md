# 템플릿 메서드

공통 로직은 부모가 처리, 세부 단계는 자식이 처리

## 예시

파일을 읽고 처리하는 로직

- 파일 열기 -> 데이터 파싱 -> 데이터 검증 -> 데이터 저장 -> 파일 닫기
- 여기서 파싱 방식만 다름
- 이럴 때 템플릿 메서드 사용

부모 클래스가 전체 흐름을 정의

```java
public abstract class DataProcessor {
    public final void process() {
        openFile();
        parseData();
        validateData();
        saveData();
        closeFile();
    }

    private void openFile() {
        System.out.println("파일 열기");
    }

    protected abstract void parseData();

    protected void validateData() {
        System.out.println("기본 검증");
    }

    private void saveData() {
        System.out.println("데이터 저장");
    }

    private void closeFile() {
        System.out.println("파일 닫기");
    }
}
```

여기서 모든 작업들을 모아놓은 process()가 템플릿 메서드 -> 전체 로직을 정의(순서, 작업 종류 등)

final키워드를 사용하여 자식 클래스에서 전체 흐름을 바꾸지 못하도록 고정

-> 위 코드에서 볼 수 있듯이

- 부모 클래스에선
	- 공통 작업 내용들 구현
	- 작업 순서 고정
- 자식 클래스에선
	- 일부 클래스 구현(파싱 클래스)

결제 처리도 비슷하게 구현

## 메서드 선언부 뜯어보기

위 코드에서 메서드들의 선언부를 잘 보면

- process() -> public final
- openFile() -> private
- parseData() -> protected abstract
- validateData() -> protected
- saveData() -> private
- closeFile() -> private

- process는 외부 클라이언트가 호출할수 있어야 하면서 동시에 자식의 오버라이딩은 막아야되기 때문에 public final 이해 감
- openFile, saveData, closeFile은 로직 자체를 고정시켜서 자식이 임의로 로직을 고치지 못하게 구현한 메서드이기 때문에 private으로 선언한거 이해 감
- parseData는 템플릿 메서드, 즉 종류에 따라 로직이 달라져야하는 메서드이기 때문에 추상 메서드로 선언한거 이해 감
- 그럼 validateData는 왜 protected지?
	- 데이터의 종류에 따라 검증 방법이 달라진다면? -> 이런 경우, 특정 자식 클래스에서만 로직 수정, 즉 오버라이딩을 허용할 수 있게 이렇게 선언
	- 이는 내부 정책에 따라 달라짐
