# 옵저버

어떤 객체의 상태가 바뀌었을 때, 그 객체를 구독하고 있는 여러 객체에게 자동으로 알림을 보내는 행동 패턴

## 예시

주문이 완료되면 다음과 같은 일들이 일어나야 한다고 하자

- 주문 완료 이메일 발송
- 재고 차감
- 쿠폰 사용 처리
- 관리자 알림
- 추천 시스템 이벤트 기록

이것들을 하나의 클래스에서 처리하게 한다? -> 단일 책임 원칙 위배

따라서 옵저버 패턴 도입

## 구성 요소

- subject
	- 상태 변경이나 이벤트 발생을 관리하는 객체
	- Observer 목록을 가지고 알림을 보냄
- observer
	- subject의 변경을 통지받는 인터페이스
- concreteObserver
	- 실제로 알림을 받고 동작하는 구독자
- Event
	- 알림 트리거

## 왜 이렇게 해야되지? 그냥 직접 부르면 안되나?

- 느슨한 결합을 위해서

	-> 옵저버 없이 직접 호출하면 비지니스 로직이 모든 후속처리 객체를 다 알고 호출해야됨 -> 높은 결합도

	따라서 후속 작업(알림 구독자)가 추가될 때마다 코드가 바뀌어야됨
