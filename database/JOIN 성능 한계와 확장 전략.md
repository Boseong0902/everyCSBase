# JOIN 성능 한계와 확장 전략

## 관계형 DB에서의 무지성 JOIN의 한계

- JOIN이 많아질수록 성능이 저하되고, 수직적인 확장만 가능해져서 확정성이 제한된다
- 특히 대규모 환경에서 복잡한 JOIN은 병목의 원인이 된다

이를 극복하기 위해서는 다음 전략을 고려한다.

## Redis 캐시

Redis 캐시를 활용하여 읽기 성능을 향상시킨다.

반복적인 읽기가 많고, 같은 데이터를 자주 조회하는 경우 캐시를 통해 DB 접근 횟수를 줄일 수 있다.

관련 개념: [[Redis 캐시]]

## 비정규화

관계를 비정규화하여 조회 속도를 확보한다.

정규화는 무결성과 정합성에는 유리하지만, 읽기 시 JOIN 비용이 커질 수 있다.

조회 성능이 더 중요하고 즉각적인 정합성 우선순위가 낮은 경우 역정규화를 고려할 수 있다.

관련 개념: [[정규화와 역정규화]]

## 샤딩

하나의 테이블 혹은 데이터베이스를 특정 기준에 따라 나눠서 다른 DB서버에 저장하는 방식

이를 통해 한 DB에서 모든 데이터를 저장하면 생기는 성능적 한계와 읽기 및 쓰기 부하를 분산시켜 확장성을 확보할 수 있다

## 샤딩의 문제

- 조인 불가능
  - 1번 서버의 테이블과 2번 서버의 테이블을 어떻게 조인할건데?
- 트랜잭션 파편화
  - 2PC, 2 Phase Commit
  - 성능 저하
- 인플루언서 데이터로 인한 특정 서버 부하 가중
  - 분산 이유 없음

## CQRS

데이터 수정 명령과 조회 명령을 분리하여 처리하는 방법

POST, PUT, DELETE와 같은 데이터 변경 요청과 GET 같은 조회 요청을 분리한다.

- 읽기는 캐시 기반
- 쓰기는 데이터 무결성에 초점

## CAP 고려

- Consistency
- Availability
- Partition Tolerance

예:

- CP: 송금 시스템
- AP: SNS
