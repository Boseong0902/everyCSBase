# Redis 캐시

Redis는 인메모리 데이터베이스

인메모리 데이터베이스이다보니 빠르기는 하지만 휘발성이라는 어느 정도의 위험성과 용량 제한이 걸려있다.

따라서 어떤 데이터를, 얼만큼의 데이터를 저장하고 어느 데이터를 캐시에서 우선적으로 제거해야하는지 등에 대한 전략을 잘 세워야한다

## Cache Hit & Cache Miss

캐시에 데이터가 있으면 Cache Hit

캐시에 데이터가 없어서 DB까지 조회해야 하면 Cache Miss

## 데이터 정합성

캐시와 데이터베이스 두 곳에서의 데이터 정합성 문제가 발생할 수 도 있다

데이터의 정합성을 높이고 Cache Miss Rate을 낮추기 위한 캐시 읽기 전략, 캐시 쓰기 전략을 잘 가져가야 한다

## 캐시 읽기 전략

## Look Aside 패턴

Cache Aside

- 데이터를 찾을 때 우선 캐시에 저장된 데이터가 있는지를 우선적으로 확인
- 만일 캐시 미스나면 DB에서 직접 조회함
- 반복적인 읽기가 많은 호출에 적합
- 캐시와 DB가 분리되어 가용되기 때문에 원하는 데이터만 별도로 구성하여 캐시에 저장
- Redis가 다운되더라도 DB에서 데이터를 가져올 수 있어서 서비스 자체는 잘돌아감
- 다만 느려질 뿐

이렇게 분리되어 작동하기 때문에 캐시와 데이터베이스 사이에 데이터 정합성 문제가 발생할 가능성이 비교적 높다.

## Read Through 패턴

- 서버는 캐시로만 요청을 보낸다
- 모든 요청을 캐시 서버로 위임하기에 서버의 부담이 적긴 하지만,
- 캐시 서버가 다운될 경우, 서비스 전체적으로 문제가 발생한다

## 캐시 쓰기 전략

## Write Back 패턴

- 데이터들을 DB에 바로 쓰지 않고, 캐시에 모아서 일정 주기 배치 작업을 통해 DB에 반영
- 캐시와 DB동기화를 비동기적으로 처리
- DB에 직접 쓰는 쿼리 횟수 자체를 줄일 수 있기에 write가 빈번할 때 사용
- 데이터 정합성 확보

## Write Through 패턴

- 데이터를 캐시에 업데이트하는 모든 요청 마다 데이터베이스에도 업데이트
- 해당 동기화 작업은 마찬가지로 캐시 서버에 위임
- 따라서 데이터 정합성의 방면에선 최고
- 하지만 다른 방식에 비해 느림

## Write Around 패턴

- DB에만 저장하기 때문에 Write Through방식에 비해 훨씬 빠름
- 캐시 미스가 발생하는 경우에만 DB와 캐시를 동기화시킴
- 정합성 측면에서 약함

## 조합

## Cache Aside + Write Through

- 읽기: 앱 -> 캐시 미스 -> 앱이 DB조회 -> 앱이 캐시에 저장 -> 반환
- 쓰기: 앱 -> 캐시 쓰기 -> DB 쓰기 -> 완료

장점:

- 캐시와 DB항상 동기화
- 캐시 장애 시 앱이 DB로 직접 풀백 가능
- 데이터를 앱이 가공해서 캐시에 저장 가능

단점:

- 쓰기마다 캐시 + DB 두 번의 쓰기가 필요
- 쓰기 지연
- Cache Stampede위험
- 읽히지 않을 데이터도 캐시에 올라감

사용:

사용자 프로필, 상품 정보처럼 읽기·쓰기가 모두 빈번하고 정합성이 중요한 일반 OLTP 서비스

## Cache Aside + Write Behind

- 읽기: 앱 -> 캐시 Miss -> 앱이 DB 조회 -> 앱이 캐시 저장 -> 반환
- 쓰기: 앱 -> 캐시 쓰기 -> 응답
- 백그라운드에서 배치로 DB 반영

장점:

- 쓰기 응답 빠름
- 동일 키 반복 쓰기 시 DB 부하 감소
- 캐시 장애 시 DB로 직접 풀백 가능

단점:

- DB 반영 전 캐시와 DB불일치 구간 존재
- DB 쓰기 실패 시, 데이터 유실 위험
- Cache Stampede위험

사용:

좋아요 수, 조회수, 게임 점수처럼 쓰기가 폭발적으로 많고 수초간의 불일치가 허용되는 서비스

## Cache Aside + Write Around

- 읽기: 앱 -> 캐시 Miss -> 앱이 DB 조회 -> 앱이 캐시 저장 -> 반환
- 쓰기: 앱 -> DB에만 쓰기 -> 완료
- 캐시 무효화 or 무시

장점:

- 캐시 오염 없음
- 안읽을 데이터는 캐시에 안올라감
- 쓰기 로직 단순화
- 캐시 장애 시 DB직접 풀백 가능

단점:

- 쓰기 후 첫 읽기는 반드시 캐시 미스
- 캐시 warm up으로 해결 가능
- Cache Stampede 위험

## Read Through + Write Through

- 읽기: 앱 -> 캐시 Miss -> 캐시가 DB 조회 -> 캐시 저장 -> 반환
- 쓰기: 앱 -> 캐시 쓰기 -> 캐시가 DB 쓰기 -> 완료

장점:

- 캐시와 DB사이의 정합성은 완벽히 일치
- stampede 방어를 캐시 레이어에서 가능

단점:

- 캐시 장애 시 서비스 중단
- 매 쓰기 요청마다 DB, 캐시 두번의 작업이 발생
- 쓰기 지연
- 데이터 가공 저장 어려움

## Read Through + Write Behind

- 읽기: 앱 -> 캐시 Miss -> 캐시가 DB 조회 -> 캐시 저장 -> 반환
- 쓰기: 앱 -> 캐시 쓰기 -> 응답
- 캐시 레이어가 배치로 DB 반영

장점:

- 쓰기 성능 좋고
- DB부하 낮음

단점:

- 캐시 장애시 끝
- 정합성 불일치 구간도 존재
- 구현 복잡도 높음

## Read Through + Write Around

- 읽기: 앱 -> 캐시 Miss -> 캐시가 DB 조회 -> 캐시 저장 -> 반환
- 쓰기: 앱 -> DB에만 쓰기 -> 완료

정합성 보통 / 캐시 효율 높음 / 장애 내성 없음

## 전략 선택 기준

- 장애 허용 여부: cache aside vs read through
- 정합성 반영 지연 시간의 허용 정도: write through vs write behind vs write around
- 쓰기 빈도: write through vs write behind
- 캐시 오염: write around로 해결

## Redis Cache Stampede

여러 이유로 동일한 데이터들에 대해서 디비에 불필요한 접근 횟수가 늘어나는 문제.
