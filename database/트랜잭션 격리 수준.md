# 트랜잭션 격리 수준

## 공유락 & 배타락

## 공유 락

읽기 작업을 할 시에 거는 락

나 이거 읽고있으니까 수정하지 마

트랜잭션 격리 수준으로 오직 공유락의 유지기간을 제한함

하지만, MySQL, PostgreSQL의 경우, 읽기 연산은 MVCC로 작동해서 스냅샷을 읽기 때문에 공유락을 획득하진 않는다

## 배타 락

쓰기 작업할 때 거는 잠금

나 이거 수정하고 있으니까 아무도 읽지도, 수정하지도 마

따라서 격리 수준과는 무관하게 이 락이 걸린 자원에 대해선 공유락, 배타락 전부 다 획득이 불가하다

따라서 격리 수준의 영향을 받지 않는다

## MVCC

공유락 만으로 동시성을 제어하면, 읽기가 쓰기를 차단하여 성능이 떨어진다

따라서 대체로 사용하는 방법이 MVCC(MySQL, PostgreSQL)

데이터 변경 시, 이전 버전을 별도의 공간에 보관하고, 각 트랜잭션은 자신의 시작 시점에 맞는 버전의 데이터를 읽는다

이를 위해서 각 버전을 스냅샷 형태로 관리하고, 해당 스냅샷에 대한 메타데이터를 붙여서 같이 관리함
