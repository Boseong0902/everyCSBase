# 트랜잭션 격리 수준

트랜잭션 격리 수준은 동시에 실행되는 트랜잭션들이 서로의 중간 상태를 어디까지 볼 수 있는지 정하는 규칙이다.

핵심은 구현 방식에 따라 다르게 이해해야 한다.

- 락 기반 DB: 공유락을 언제 잡고 언제 해제할지로 읽기 격리를 조절한다.
- MVCC 기반 DB: 각 트랜잭션이나 SQL문이 어떤 시점의 스냅샷을 볼지로 읽기 격리를 조절한다.

MySQL InnoDB와 PostgreSQL은 일반적인 읽기에서 공유락을 잡기보다 MVCC 스냅샷을 읽는다. 그래서 "공유락을 MVCC로 대체한다"는 말은 구현 관점의 설명이고, 격리 수준 자체는 "어떤 이상 현상을 허용하거나 막는가"로 이해해야 한다.

## 공유락과 배타락

## 공유락

공유락은 읽기 작업을 할 때 거는 락이다.

의미는 "나 이 데이터를 읽고 있으니 수정하지 말라"에 가깝다.

락 기반 DB에서는 격리 수준이 공유락의 유지 기간에 직접 영향을 준다.

- Read Uncommitted: 공유락을 잡지 않거나 사실상 읽기 보호를 하지 않는다.
- Read Committed: 읽는 순간에만 공유락을 잡고 바로 해제한다.
- Repeatable Read: 한번 잡은 공유락을 트랜잭션 종료까지 유지한다.
- Serializable: 공유락뿐 아니라 범위락까지 사용해서 조건에 맞는 새로운 행의 삽입까지 막는다.

## 배타락

배타락은 쓰기 작업을 할 때 거는 락이다.

의미는 "나 이 데이터를 수정하고 있으니 다른 쓰기나 잠금 읽기는 기다리라"에 가깝다.

다만 MVCC 기반 DB에서는 배타락이 걸린 행도 일반 `SELECT`로는 이전 커밋 버전을 읽을 수 있다. 즉, 배타락이 항상 모든 읽기를 막는 것은 아니다. 충돌하는 것은 주로 다른 쓰기 작업이나 `SELECT ... FOR UPDATE` 같은 잠금 읽기다.

따라서 배타락은 격리 수준을 이해할 때 "쓰기 충돌을 막는 장치"로 보고, 일반 읽기 일관성은 MVCC 스냅샷과 격리 수준으로 따로 봐야 한다.

## MVCC

MVCC는 Multi Version Concurrency Control의 약자다.

공유락만으로 동시성을 제어하면 읽기가 쓰기를 막고, 쓰기가 읽기를 막아서 처리량이 떨어진다. MVCC는 데이터가 변경될 때 이전 버전을 보관하고, 트랜잭션이 자신에게 허용된 시점의 버전을 읽게 해서 읽기와 쓰기의 충돌을 줄인다.

정리하면 다음과 같다.

- 읽기 작업 이상: 어떤 스냅샷을 읽을지 정해서 방어한다.
- 쓰기 작업 이상: 배타락으로 충돌하는 쓰기를 블로킹한다.
- 직렬화 이상: Serializable 수준에서 충돌을 감지해 대기시키거나 커밋 시점에 실패시킨다.

MySQL과 PostgreSQL은 모두 MVCC를 사용하지만, 옛 버전을 관리하는 방식은 다르다. 이 차이는 [[MySQL과 PostgreSQL 차이]]에서 같이 보면 좋다.

## 동시성 이상 현상

## Dirty Read

Dirty Read는 커밋되지 않은 수정 중인 데이터를 다른 트랜잭션이 읽는 현상이다.

예를 들어 T1이 사용자 잔액을 1000원에서 500원으로 바꿨지만 아직 커밋하지 않았다고 하자. 이때 T2가 500원을 읽었는데, 이후 T1이 롤백하면 T2는 실제로 존재한 적 없는 값을 읽은 셈이 된다.

## Non-Repeatable Read

Non-Repeatable Read는 하나의 트랜잭션 안에서 같은 행을 여러 번 읽었는데, 읽을 때마다 값이 달라지는 현상이다.

예를 들어 T1이 주문 `id=1`의 상태를 읽었을 때 `READY`였다. 그 사이 T2가 같은 주문 상태를 `PAID`로 바꾸고 커밋했다. T1이 같은 행을 다시 읽었을 때 `PAID`가 보이면, T1 입장에서는 같은 트랜잭션 안에서 같은 행의 값이 반복 가능하지 않다.

## Phantom Read

Phantom Read는 하나의 트랜잭션 안에서 같은 조건의 범위 조회를 여러 번 했는데, 기존에 없던 행이 생기거나 사라지는 현상이다.

예를 들어 T1이 `amount >= 10000`인 주문 목록을 조회해서 3건을 봤다. 그 사이 T2가 조건에 맞는 주문을 새로 삽입하고 커밋했다. T1이 같은 조건을 다시 조회했을 때 4건이 보이면 phantom read가 발생한 것이다.

Non-Repeatable Read가 "이미 읽은 같은 행의 값이 바뀌는 문제"라면, Phantom Read는 "같은 조건에 걸리는 행 집합이 바뀌는 문제"다.

## 격리 수준별 동작

| 격리 수준 | Dirty Read | Non-Repeatable Read | Phantom Read | 핵심 동작 |
| --- | --- | --- | --- | --- |
| Read Uncommitted | 발생 가능 | 발생 가능 | 발생 가능 | 커밋되지 않은 변경까지 읽을 수 있다. |
| Read Committed | 방지 | 발생 가능 | 발생 가능 | 각 SQL문이 실행될 때마다 커밋된 데이터 기준으로 읽는다. |
| Repeatable Read | 방지 | 방지 | 표준상 발생 가능, DBMS별 구현 차이 있음 | 트랜잭션 안에서 같은 스냅샷을 반복해서 읽는다. |
| Serializable | 방지 | 방지 | 방지 | 결과가 트랜잭션을 순서대로 실행한 것처럼 보장된다. |

## Read Uncommitted

Read Uncommitted는 가장 낮은 격리 수준이다.

커밋되지 않은 데이터까지 읽을 수 있으므로 Dirty Read가 발생할 수 있다. Dirty Read를 허용한다는 것은 그보다 약한 일관성을 가지므로 Non-Repeatable Read와 Phantom Read도 막지 못한다.

락 기반으로 보면 읽기 시 공유락을 잡지 않는 수준이다. MVCC 기반 DB에서는 구현이 다를 수 있다. PostgreSQL은 `READ UNCOMMITTED`를 지정해도 실제로는 `READ COMMITTED`처럼 동작한다.

## Read Committed

Read Committed는 커밋된 데이터만 읽는 수준이다.

Dirty Read는 막는다. 하지만 하나의 트랜잭션 안에서도 SQL문마다 새 스냅샷을 볼 수 있으므로, 중간에 다른 트랜잭션이 커밋한 변경이 다음 조회에 반영될 수 있다. 그래서 Non-Repeatable Read와 Phantom Read는 발생할 수 있다.

락 기반으로 보면 읽는 순간에 공유락을 잡고 바로 해제하는 방식에 가깝다. MVCC 기반으로 보면 각 SQL문마다 스냅샷을 새로 잡는 방식이다.

PostgreSQL의 기본 격리 수준이다.

## Repeatable Read

Repeatable Read는 하나의 트랜잭션 안에서 같은 데이터를 반복해서 읽을 수 있게 하는 수준이다.

Dirty Read와 Non-Repeatable Read를 막는다. MVCC 기반으로 보면 트랜잭션에서 사용할 스냅샷을 고정하고, 같은 트랜잭션 안에서는 그 스냅샷을 계속 읽는다. 그래서 다른 트랜잭션이 중간에 커밋해도 현재 트랜잭션의 일반 읽기 결과는 바뀌지 않는다.

락 기반으로 보면 한번 잡은 공유락을 트랜잭션 끝까지 유지해서 이미 읽은 행의 수정을 막는다. 하지만 단순히 행 단위 공유락만 유지하면, 조건 범위 안에 새 행이 삽입되는 Phantom Read까지는 막지 못한다.

DBMS별로는 차이가 있다.

- MySQL InnoDB의 기본 격리 수준이다. 일반 consistent read는 트랜잭션 스냅샷을 사용하고, 잠금 읽기나 범위 조건에서는 gap lock, next-key lock으로 많은 phantom read 상황을 막는다. 이 내용은 [[인덱스]]의 범위 탐색과도 연결된다.
- PostgreSQL의 Repeatable Read는 snapshot isolation으로 동작하며 일반적인 phantom read를 허용하지 않는다. 다만 모든 직렬화 이상을 막는 것은 아니므로 완전한 직렬 실행 보장이 필요하면 Serializable을 사용해야 한다.

## Serializable

Serializable은 가장 강한 격리 수준이다

동시에 실행했지만, 결과만 보면 둘 중 하나가 먼저 실행되고 그다음 다른 하나가 실행된 것처럼 보여야 한다 라는 원칙의 격리수준이다. 여기서 이 트랜잭션들의 순서를 정하는걸 직렬화라고 한다.

트랜잭션들이 동시에 실행되더라도 결과가 어떤 순서로 하나씩 실행된 것처럼 보여야 한다. Dirty Read, Non-Repeatable Read, Phantom Read를 모두 막는다.

락 기반 DB에서는 공유락과 범위락을 강하게 사용해서 조건 범위에 새 행이 끼어드는 것까지 막는다. MVCC 기반 DB에서는 스냅샷만으로 처리하지 않고, 직렬화 불가능한 충돌을 감지해서 대기시키거나 커밋 시점에 트랜잭션을 의도적으로 실패시킬 수 있다.

동시성은 가장 낮고 비용은 가장 크지만, 금융 정산처럼 결과의 직렬성이 중요한 작업에서는 필요할 수 있다.

## 정리

격리 수준을 외울 때는 다음 순서로 보는 것이 좋다.

1. Read Uncommitted는 커밋 전 데이터까지 보므로 Dirty Read부터 발생한다.
2. Read Committed는 커밋된 데이터만 보므로 Dirty Read는 막지만, SQL문마다 보는 시점이 달라질 수 있다.
3. Repeatable Read는 트랜잭션 안의 읽기 시점을 고정해서 같은 행의 반복 읽기를 보장한다.
4. Serializable은 같은 조건 범위에 새 행이 들어오는 문제까지 막고, 전체 결과가 직렬 실행처럼 보이게 한다.

면접에서는 "MySQL과 PostgreSQL은 MVCC를 쓰기 때문에 일반 읽기가 항상 공유락을 잡는 방식으로 동작하지 않는다"는 점까지 같이 말해야 설명이 정확해진다.
