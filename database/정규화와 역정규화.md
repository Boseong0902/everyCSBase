# 정규화와 역정규화

## 목표

데이터의 중복을 줄이고, 무결성을 유지하는 구조로 설계해보자

결국 정규화란 데이터의 무결성, 정합성을 위해서 테이블을 여러 테이블로 분할하는 과정이다

따라서 읽기 연산에 있어서 기존엔 단순 `SELECT` 문으로 가능했던 연산이 `JOIN`이 필요하게 될 수도 있다

이러한 트레이드오프를 인지하고 반영해야한다

## 1NF

## 목적

다중 값 제거

## 왜?

- 삽입 이상
- 검색 이상
- 갱신 이상

## 2NF

## 목적

기본 키가 복합 키일 때, 일부 키에만 의존하는 종속 관계를 없애는 게 목적

## 왜?

- 삽입 이상
- 갱신 이상
- 삭제 이상

## 3NF

## 목적

이행적 함수 종속 관계 제거

즉, 키가 아닌 컬럼에 종속되는 관계가 있을 경우

## 왜?

- 갱신 이상
- 삽입 이상
- 삭제 이상

## 언제 정규화가 더 필요한가?

데이터의 수정이 많이 일어날 때

- 쓰기 > 읽기
- 같은 데이터를 여러 팀, 서비스가 수정한다
- 데이터 정합성이 제일 중요하다
- 스키마가 앞으로 자주 바뀔 가능성이 높다
- ORM 캐시, 애플리케이션 레이어에서 조인 비용 흡수 가능하다

## 언제 역정규화가 필요한가?

데이터의 수정보다는 읽기 작업에 대한 성능 개선을 위할 때

- 읽기가 압도적으로 많다
- 조인 대상 테이블이 수천만 행 이상
- 실시간 집계 대시보드가 중요하다
- 데이터 분석 전용 DB
- 쓰기는 배치로만 발생하고, 즉각적인 정합성의 우선순위가 낮을 때
