# 트랜잭션

데이터베이스의 상태를 변경시키기 위해 수행하는 작업 단위

## 특징

## Atomicity

원자성

반영 되거나 안되거나

## Consistency

일관성

무결성 제약조건이 깨지면 안된다

즉, DB의 모든 트랜잭션의 실행으로 인해 무결성 제약조건이 유효한 상태에서 무결성 제약조건이 유효한, 값만 바뀐 또 다른 상태로 바뀌어야 한다

## Isolation

격리성

각 트랜잭션은 다른 트랜잭션과 격리되어서 실행되어야 한다

## Durability

지속성

트랜잭션의 커밋이 영구적으로 반영되어야함

## Commit & Rollback

## Commit

하나의 트랜잭션이 성공적으로 끝나, 이를 확정하는 연산

커밋이 수행되면 트랜잭션의 변경사항이 영구적으로 반영된다

## Rollback

트랜잭션이 비정상적으로 종료되어 ACID가 깨진 경우, 실행 이전 상태로 되돌리는 연산

## Savepoint

트랜잭션 내부에 중간 저장 지점을 설정할 수 있다

즉 전체 롤백을 안하고 특정 지점까지만 되돌릴 수도 있다

## 트랜잭션 반영 프로세스

`UPDATE -> COMMIT -> 디스크 반영`

## MySQL - InnoDB

- undo log: 변경 전 옛 값
- redo log: 변경 후 되어야 하는 값
- WAL원칙: Write Ahead Logging, 무조건 로그부터

## UPDATE 실행 시

1. 실행 쿼리를 메모리로 로드
2. Undo Log기록 -> 롤백 & MVCC 대비
3. 메모리에서 값을 수정
4. Redo Log에 수정 사항을 기록

메모리랑 디스크는 값이 다름. dirty page.

## COMMIT 실행

1. Redo Log를 디스크에 강제로 flush
2. 커밋 완료 응답을 반환

## 나중에 비동기적으로 디스크에 반영

더티 페이지를 디스크 테이블에 반영

장애 발생 시점이 커밋 전이냐 후이냐로 갈림

- 커밋 전: 아직 Redo Log 작성 안됨 -> Undo Log 참고해서 롤백 진행
- 커밋 후: Redo Log가 생성되어있으므로, Redo Log 참고해서 디스크에 다시 반영

커밋이 안된 상태이더라도 메모리가 꽉차면 디스크에 저장을 하고, 메모리에선 삭제를 해버리는 작업이 발생할 수 있다.

따라서 디스크에 해당 데이터가 이미 반영이 되어있을수도 있으므로 undo log 기반의 롤백이 필요할 수 있다.

## PostgreSQL

PSQL또한 WAL원칙을 따름

- xid: 자원 수정을 하는 트랜잭션 ID
- xmin: 해당 튜플을 만든 트랜잭션의 xid
- xmax: 해당 튜플을 죽인 트랜잭션의 xid
- CLOG: 각 xid별 커밋 되었는지 여부를 기록

## UPDATE 실행 시

1. 새 버전의 튜플에 xmin을 같이 써둠
2. 구 버전의 튜플도 해당 테이블에 두지만, xmax를 같이 써둠
3. 따라서 신, 구 값이 모두 테이블에 공존하지만, 데이터의 유효 여부가 다름

## COMMIT 실행 시

1. Redo Log를 디스크에 flush
2. CLOG에 해당 트랜잭션의 xid를 commited로 수정

## 나중에 비동기적으로 디스크에 반영

1. 더티 페이지를 디스크 테이블에 반영
2. vaccuum프로세스가 구버전 튜플 청소
