# MySQL과 PostgreSQL 차이

## 연결 구조

PostgreSQL은 프로세스 단위의 연결이므로, 각 연결마다 비용이 크다

따라서 많은 연결을 못버티므로 앱 레벨의 풀(HikariCP)만으론 부족하다

이를 압축시켜줄 외부 풀러가 필요하다

대표적으로 PgBouncer가 필요하다.

예:

- 인스턴스 10개가 각각이 100개의 커넥션 풀 가용
- 총 1000개의 커넥션풀
- PostgreSQL 죽음
- 이 1000개의 커넥션 풀을 100개로 압축시켜주는 외부 풀러가 필요

## PostgreSQL이 더 우세한 것들

## 트랜잭션 DDL

PostgreSQL은 스키마 변경도 트랜잭션 단위로 하고, 실패 시 롤백 가능

MySQL은 DDL을 만나면 진행중이던 트랜잭션 강제 커밋

따라서 롤백 불가능

즉 마이그레이션의 안정성 측면에서 더 확실하게 보장된다

## Materialized View

일반 뷰는 쿼리를 저장하고 매 요청마다 계산을 다시함

Materialized View는 계산 결과를 캐싱함

## 특수 인덱스

역인덱스나 부분 인덱스 같은 경우, MySQL보다 우세하거나 MySQL에선 아예 제공 안한다

## MySQL은 언제?

읽기 작업이 많을 때

우선 MySQL은 스레드 기반이라서 커넥션 연결, 유지 비용이 비교적 싸다

## 클러스터링 인덱스

모든 데이터들이 PK에 대해서 정렬되어있음

따라서 PK인덱싱만 따라가면 별도의 추가 연산 없이 해당 데이터 찾을 수 있음

나머지 인덱스는 인덱스를 따라가면 해당 데이터의 메타데이터(위치정보)를 찾고, 위치정보를 갖고 힙에서 추가 조회가 필요함

PostgreSQL은 클러스터링 인덱스를 제공하지 않음

## MVCC 관리 방식

PostgreSQL은 죽은 튜플들이 청소되기 전까진, 다른 튜플들과 동일하게 인덱스에 타짐

따라서 검색에도 느리고, 이게 유효한 튜플인지에 대한 검사도 따로 진행해야됨

따라서 읽기 작업에 있어서 MySQL이 더 유리한 부분이 명확하다
