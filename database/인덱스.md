# 인덱스

## 목적

추가적인 쓰기 작업과 저장 공간을 활용하여 데이터베이스 테이블의 검색 속도를 향상시키기 위한 자료구조

## 언제?

`explain` 쿼리를 돌렸을 때, 풀스캔이 포착되면 인덱싱 고려

## 핵심

데이터베이스 안의 레코드를 처음부터 풀스캔하지 않고, B+Tree로 구성된 구조에서 index 파일 검색으로 속도를 향상시키는 기술이다

## 인덱스 종류

## 클러스터형 인덱스

- 테이블당 하나만 생성 가능
- 데이터의 물리적 저장 순서 자체를 인덱스 키 순서로 정렬
- Primary Key를 설정하면 자동으로 클러스터형 인덱스가 생성된다
- 물리적으로 인접한 데이터를 연속적으로 읽을 수 있기 때문에 이 인덱스를 적용한 컬럼을 기준으로 한 범위 검색에 특히 유리하다

## 비클러스터형 인덱스

- 테이블당 여러 개 생성 가능
- 데이터의 물리적 순서를 변경하지 않고, 별도의 인덱스 페이지에 논리적 순서를 저장한다
- 리프노드에 실제 데이터가 아닌, 데이터의 위치를 가리키는 포인터가 저장된다

## 복합 인덱스

- 두 개 이상의 컬럼을 조합하여 하나의 인덱스로 생성한다
- 선두 컬럼을 기준으로 먼저 정렬, 후행 컬럼을 기준으로 그 다음 정렬

예: `user_id`와 `order_date`를 기준으로 많이 탐색한다고 치자

- `user_id`를 기준으로 먼저 정렬
- 동일한 `user_id`의 경우, `order_date`를 기준으로 정렬

## 커버링 인덱스

쿼리에서 요구하는 모든 컬럼이 인덱스 안에 포함되어있는 경우

이런 경우, 굳이 테이블에 접근 안해도 되므로, I/O가 크게 줄어든다

EXPLAIN으로 쿼리 실행 계획을 찍어보면

- MySQL: `Using index`
- PostgreSQL: `Index Only Scan`

느린 쿼리에 대해서 실행 계획을 찍어봤을 때 발견된 병목지점에 대한 입출력 비용을 줄이기 위해서 일부러 해당 컬럼을 복합 인덱스로 걸수도 있다

## 인덱스 자료구조

## B-Tree

- 탐색 시간: O(log N)
- 모든 노드에 키와 데이터를 함께 저장하며, 각 노드가 포인터 역할도 수행

## B+Tree

- 리프 노드에만 실제 데이터를 저장
- 내부 노드는 탐색을 위한 키와 포인터 역할을 수행
- 리프 노드끼리 링크드리스트로 연결되어있어서 범위 검색에 효율적

## Hash Table

- 탐색, 삽입, 삭제 연산에 대해서 O(1) 시간 복잡도
- 하지만 순차탐색, 구간 탐색등에는 O(N)
- 따라서 오로지 등호 연산만 사용하는 컬럼에 대해서 사용
- Redis나 memcached같이 키-값 저장소
- MySQL에서도 memory(heap) 엔진이 해시 인덱스를 사용한다

## 인덱스 스캔 방식

## Index Range Scan

링크드리스트로 연결된 리프노드에서 구간탐색 수행

## Index Full Scan

테이블 풀스캔보단 낫다

기본적으로 인덱스는 일단 정렬은 돼있고, 커버링 인덱스로 해결될 수도 있다

## Index Unique Scan

등호 스캔

해시테이블에서 사용 -> 가장 빠르긴 함(O(1))

## 인덱싱의 장단점

## 장점

- 검색속도 향상 O(N) -> O(log N)
- 정렬 성능 향상

## 단점

- 인덱스를 위한 추가 저장공간 필요
- 인덱스된 필드에서 데이터를 업데이트하거나, 레코드를 추가/삭제 시 성능이 떨어진다
- 타겟 레코드를 찾는 시간 자체는 줄어들지만, 찾고 나서의 insert, update, delete의 시간이 훨씬 더 늘어난다
- 인덱스에서 해당 레코드의 키의 정확한 위치에 삽입을 해야되니까
- 따라서 DML이 자주 발생하는 컬럼에 대해서는 인덱스 사용을 지양해야된다

## 인덱스는 누구한테?

## 우선순위

- 우선 explain쿼리 -> 풀스캔 여부 확인
- PK, FK, JOIN칼럼
  - PK는 자동 클러스터형 인덱스가 생긴다
  - FK, JOIN칼럼은 인덱스가 없으면 조인할때마다 풀스캔이 발생
- WHERE절에 자주 등장하는 칼럼
- 카디널리티가 높은 칼럼
  - 인덱스로 필터링되어서 제외되는 레코드들이 늘어나니까
- ORDER BY, GROUP BY
  - 이미 정렬 되어있는 인덱스가 존재한다면, ORDER BY, GROUP BY에서의 정렬 비용이 삭제되니까 효율적

## 피해야되는 우선순위

- DML빈번
- 카디널리티가 낮은 칼럼
- 데이터가 적은 테이블
  - 테이블의 행이 수백건 수준이면, 풀스캔이 오히려 빠르다
  - 인덱스 트리를 타는게 오히려 오버헤드가 더 크다

## DML 발생 시 인덱스 동작

## INSERT

데이터 삽입 해보는데, 해당 데이터가 들어갈 블록이 꽉차있으면 index Split을 수행

이러면 그 key에 해당하는 데이터들에 대한 DML이 블로킹되어서 대기상태가 된다

## DELETE

Table과 Index의 데이터 정합성의 차이

- Table: 데이터가 삭제되고 공간이 재사용 가능
- Index: 데이터가 삭제되지 않고, 사용 안됨으로 표시만 남김

이로 인해 Table의 데이터 수와 Index의 데이터 수가 불일치한다

이는 주기적으로 Purge Thread가 동작하면서 죽은 노드를 정리하지만, 그 사이에 죽은 노드가 쌓여서 성능 저하로 이어지게 된다

## UPDATE

인덱스는 직접적으로 UPDATE를 지원하지 않는다

즉, DELETE + INSERT를 하는 2단계의 작업으로 분리되어서 처리된다
