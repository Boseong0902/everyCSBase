# SOLID와 디자인 패턴 면접 질문

## 디자인 패턴과 SOLID

관련 개념: [[디자인 패턴]], [[SOLID]], [[GoF]]

- 디자인 패턴이란 무엇인가?
- SOLID는 디자인 패턴인가, 설계 원칙인가?
- GoF 디자인 패턴과 SOLID는 어떤 관계인가?
- 높은 유지보수성과 확정성을 위해 설계 원칙이 필요한 이유는 무엇인가?
- 디자인 패턴을 남용하면 어떤 문제가 생기는가?

## GoF 생성 패턴

관련 개념: [[GoF]], [[싱글톤]], [[정적 팩토리 메서드와 팩토리 메서드]], [[빌더]]

- 생성 패턴은 무엇을 목적으로 하는가?
- 객체를 직접 new로 생성하면 어떤 결합 문제가 생기는가?
- 싱글톤은 어떤 상황에서 사용하는가?
- 싱글톤 구현 방식에는 무엇이 있는가?
- 이른 초기화와 지연 초기화는 어떤 트레이드오프를 가지는가?
- 지연 초기화 싱글톤에서 `synchronized`를 메서드 전체에 걸면 어떤 문제가 생기는가?
- Double-Checked Locking에서 `volatile`이 필요한 이유는 무엇인가?
- static inner class 방식이 별도 동기화 없이도 안전한 이유는 무엇인가?
- Enum 싱글톤은 다른 방식에 비해 어떤 점이 안전한가?
- 싱글톤이 강한 결합을 만들 수 있는 이유는 무엇인가?
- 싱글톤을 클래스 내부에서 직접 호출하면 테스트가 왜 어려워지는가?
- 스프링 빈의 싱글톤 관리와 직접 구현한 싱글톤은 어떤 차이가 있는가?
- 빌더 패턴은 어떤 상황에서 사용하는가?
- 빌더 패턴이 불변 객체 생성에 유리한 이유는 무엇인가?
- 정적 팩토리 메서드는 생성자에 비해 어떤 장점이 있는가?
- Simple Factory와 Factory Method Pattern의 차이는 무엇인가?
- Simple Factory가 완전한 OCP를 보장하지 못하는 이유는 무엇인가?
- 빌더 메서드가 `this`를 반환하는 구조는 어떻게 메서드 체이닝을 가능하게 하는가?
- 빌더 패턴의 단점은 무엇인가?
- Abstract Factory Pattern은 어떤 상황에서 사용하는가?

## GoF 구조 패턴

관련 개념: [[GoF]], [[어댑터]], [[프록시]], [[데코레이터]], [[스프링 AOP]]

- 구조 패턴은 무엇을 목적으로 하는가?
- 어댑터 패턴은 어떤 상황에서 사용하는가?
- 어댑터가 단순히 메서드명만 바꾸는 것이 아니라고 하는 이유는 무엇인가?
- 프록시 패턴은 무엇이며 언제 사용하는가?
- 프록시와 데코레이터의 차이는 무엇인가?
- 스프링 AOP는 프록시 패턴과 어떤 관계인가?
- 스프링에서 프록시가 만들어지려면 왜 빈으로 등록되어야 하는가?
- `@Transactional`, `@Cacheable`, `@Async`는 각각 프록시가 어떻게 개입하는가?
- 데코레이터 패턴은 어떤 상황에서 유리한가?
- 알림 채널 조합처럼 경우의 수가 폭발하는 상황을 데코레이터는 어떻게 해결하는가?
- 데코레이터 조합이 런타임에 바뀌어야 한다면 어떤 패턴과 함께 쓰는가?
- 프록시가 실제 객체 호출 전후에 넣는 대표적인 제어 로직에는 무엇이 있는가?
- JDK 동적 프록시와 CGLIB 프록시는 어떻게 다른가?
- 스프링 AOP에서 같은 클래스 내부 메서드를 호출하면 왜 프록시가 적용되지 않는가?

## GoF 행위 패턴

관련 개념: [[GoF]], [[전략]], [[템플릿 메서드]], [[옵저버]], [[이벤트 리스너와 트랜잭션]]

- 행위 패턴은 무엇을 목적으로 하는가?
- 전략 패턴은 어떤 상황에서 사용하는가?
- 어떤 신호가 보이면 전략 패턴 도입을 고려해야 하는가?
- 전략 패턴이 OCP, 다형성과 어떻게 연결되는가?
- 템플릿 메서드 패턴에서 process()를 public final로 선언하는 이유는 무엇인가?
- 템플릿 메서드에서 훅 메서드를 protected로 여는 경우는 언제인가?
- 전략 패턴과 템플릿 메서드 패턴의 차이는 무엇인가?
- 스프링에서 전략들을 `Map`으로 주입받는 방식은 어떤 이점이 있는가?
- 스프링의 `JdbcTemplate`이나 `RestTemplate`은 템플릿 메서드 패턴과 어떻게 연결되는가?
- 옵저버 패턴은 어떤 상황에서 사용하며 SRP와 어떻게 연결되는가?
- 옵저버 패턴의 구성 요소는 무엇인가?
- 후속 작업을 그냥 직접 호출하지 않고 이벤트로 분리하는 이유는 무엇인가?
- `@EventListener`와 `@TransactionalEventListener`의 차이는 무엇인가?
- TransactionPhase의 종류(BEFORE_COMMIT, AFTER_COMMIT, AFTER_ROLLBACK, AFTER_COMPLETION)는 각각 언제 사용하는가?
- BEFORE_COMMIT 단계에서 예외가 발생하면 트랜잭션은 어떻게 되는가?
- AFTER_COMMIT으로 후속 작업을 분리하면 어떤 정합성 문제가 남는가? (-> [[분산 트랜잭션]])
- AFTER_COMMIT 이후 후속 작업이 실패했을 때 어떤 보완책을 쓸 수 있는가?
- `@Async`를 함께 붙이면 후속 작업의 예외 전파는 어떻게 달라지는가?

## OCP

관련 개념: [[OCP - 개방-폐쇄]], [[interface]], [[정적 다형성과 런타임 다형성]]

- 개방-폐쇄 원칙이란 무엇인가?

- OCP를 지키는 데 인터페이스가 어떻게 사용되는가?
- 도형 예시에서 AreaCalculator가 OCP를 위반하는 이유는 무엇인가?
- SRP와 OCP의 차이는 무엇인가?

- SRP는 지켰지만 OCP는 지키지 못한 코드는 어떤 모습인가?

## LSP

- 리스코프 치환 원칙이란 무엇인가?

- 부모의 필드나 메서드 합의가 자식 클래스에서 깨지면 어떤 문제가 생기는가?

- 구현이 완료된 메서드를 아예 문법적으로 오버라이딩 금지하면 되지 않는가?
- LSP를 지키지 못하는 상속 관계는 어떤 방식으로 바꿀 수 있는가?

- 의존 관계 역전 원칙이란 무엇인가?
