# 페일오버

## 배포 후 장애의 종류

- 사람 장애
- 자연 장애(스펙 부족)

뭐 어떤 장애던 간, 장애가 발생한 서비스 컴포넌트의 장애를 인지하고, 계속해서 서비스를 이어나갈 수 있어야 된다

페일오버 -> 장애 발생 시, 서비스 컴포넌트의 장애를 인지하고 해당 기능을 대신하는 새로운 컴포넌트를 동작하게 하는 기술

## 액티브-스탠바이 페일오버

서비스를 하지 않는 대기 서버를 준비해두고, 서비스 중인 서버에 장애가 발생하면 바로 대기 서버로 대체하는 형식

## 액티브-액티브 페일오버

Scale-out된 환경에서 로드밸런서를 앞단에 두고 있는 여러 API서버들

운영 도중 서버가 바뀌는 상황은 페일오버 뿐만 아니라 무중단 배포도 마찬가지

- 무중단 배포 -> 기존 서버가 처리하던 요청은 기존 서버가 마저 처리하고 새 요청부터 새 서버로 보냄
- 즉, 서버가 처리하고 있던 기존 트래픽을 다른 서버에서 마저 처리하게 할 수 없음
- 이는 페일오버에서도 마찬가지

이를 자동화된 페일오버로 보장할 수 있음

## VIP 페일오버

### IP 주소

인터넷 망에서 컴퓨터를 찾아가기 위한 가상의 주소

### MAC 주소

컴퓨터 랜카드에 부여된 고유한 물리적 주소

### Switch

네트워크 장비이다

ARP테이블에 IP주소와 MAC주소의 매핑 정보를 저장하여, 각 요청들을 적절한 컴퓨터로 라우팅해줌

### 설명

네트워크 안에 세 대의 컴퓨터와 하나의 스위치가 연결되어있다고 가정

- 애플리케이션 서버 - 데이터를 보내는 컴퓨터
- 1번 DB - `10.0.0.10`
- 2번 DB - `10.0.0.20`

여기서 고가용성을 위해 가상 IP(VIP)인 `10.0.0.100`을 하나 더 만든다

### 평상시 상태

1번 DB서버가 자신의 랜카드에 가상 IP인 `10.0.0.100`을 임의로 등록해 놓음

스위치의 ARP테이블에는 다음과 같이 기록됨

`10.0.0.100`으로 가는 데이터들은 1번 DB서버의 MAC주소로 보내면 됨

애플리케이션 서버는 목적지를 `10.0.0.100`으로 지정해서 데이터를 보냄

스위치는 표를 보고 이 데이터를 1번 DB 서버로 보냄

### 1번 DB서버가 죽으면?

2번 DB서버는 1번 DB서버가 응답하지 않는 것을 확인하고, `(with.heartbeat_pingpong)` 자신의 랜카드에 가상 IP(`10.0.0.100`)을 등록하고,

이러면 2번 DB서버는 `10.0.0.20`과 `10.0.0.100` 두 개의 IP를 가진다

그 후, 2번 DB서버는 네트워크 망 전체에 GARP라는 특수한 패킷을 브로드캐스트 함

GARP - Gratuitous ARP -> 지금부터 `10.0.0.100`을 쓰는 컴퓨터는 나(2번 DB)이다.

내 맥 주소는 ~~~이니까 이거 ARP테이블에 `10.0.0.100`과 매핑해놔

이제 2번 DB서버로 요청이 들어오게 됨

### 장단점

클라이언트 입장에선 편하다

위 예시에서는 백엔드 애플리케이션 서버가 클라이언트가 되겠지

그랬을 때, 클라이언트는 그냥 계속해서 동일한 IP주소에 요청을 보내도 되니까 별다른 무언가를 할 필요가 없다

심지어 DB서버가 죽었는지도 모른다

GARP패킷을 브로드캐스트 해서 스위치에서의 라우팅을 바꿀 수 있어야 하기 때문에 동일한 L2네트워크 내에 있어야 된다

L2 -> 스위치 한개가 관리하는 영역

BGP프로토콜 -> 서비스를 운영하는 입장에서는 설정이 복잡하다

따라서 단점이 극명하기 때문에 AWS를 비롯한 대부분의 클라우드 서비스에선 DNS페일오버를 제공한다

## DNS 페일오버

그냥 DNS에서 해당 도메인과 매핑되는 IP주소를 바꿔주는 방식

### 평상시

- 애플리케이션 서버 1 - `10.0.0.1` <- 여기에 `myservice.com`이 매핑되어 있음
- 애플리케이션 서버 2 - `20.0.0.1`

따라서 매 요청마다 DNS는 `10.0.0.1`로 매핑해줌

그리고 클라이언트는 매 요청마다 DNS를 거치지 않기 위해서 도메인-IP 주소의 매핑 정보를 저장하고 있음 - TTL 설정 가능

### 장애 발생 시

DNS 혹은 별도의 인스턴스가 두 서버의 상태를 감시중, 서버1이 다운됨

DNS는 이제 `myservice.com`으로 들어오는 요청을 `20.0.0.1`로 매핑해줌

이러면 다 해피케이스같지만, TTL이 아직 안끝난 클라이언트 로컬은 계속해서 서버1로 요청을 보냄

그동안은 정상 응답을 받지 못함
