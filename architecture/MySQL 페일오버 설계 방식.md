# MySQL 페일오버 설계 방식

보통 Primary랑 Replica는

- 동일 vpc
- 다른 AZ

에 배치한다

## 복제 매커니즘

### GTID

트랜잭션마다 고유한 ID를 부여하여 Primary가 죽었을 때 Replica가 어느 시점까지 트랜잭션을 반영했는지를 추적.

### 반동기 복제

Primary가 트랜잭션을 처리한 후, 최소 한 대의 Replica애서 릴레이 로그에 기록했다는 ACK을 받을때까지 대기

즉, Replica 중 적어도 한 곳에는 써놓고 트랜잭션을 종료하겠다라는 말

## 실무에서의 구성 및 페일오버 도구

### InnoDB Cluster

모든 노드들간에 주기적 헬스체크 가 이루어짐

primary node가 죽으면 이에 대해서 헬스체크 가 중단됨

이를 통해 Primary가 죽음을 알고, 가장 최신 트랜잭션에 대해 커밋이 찍혀있는 Replica가 내부 노드들 사이의 투표를 통해 결정됨

라우팅은 MySQL 내부 라우터가 존재함 -> 이걸로 라우팅 변경
