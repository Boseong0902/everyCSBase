# PostgreSQL 페일오버 설계 방식

자동 페일오버가 내장되어있진 않음

Primary에서 WAL을 바이트스트림 단위로 Replica로 실시간으로 전송

- primary에선 데이터 변경 이전에 WAL 로그를 찍고 데이터 변경 수행
- primary엔 replica랑 1:1로 매핑되는 로그 전송 전용 백그라운드 프로세스가 존재
- 네트워크 TCP소켓을 통해 Replica로 보냄

트랜잭션 동기화 수준 설정 가능(동기 vs 비동기) -> (성능 vs 정합성)

## 실무에서의 구성 및 페일오버 도구

### Patroni + 분산 코디네이터

Primary 생존 여부 확인 + replica 선택

#### patroni

각 DB인스턴스 안에서 실행되는 데몬 프로세스

DB 설정 파일을 관리하고, 생존 여부를 분산 코디네이터에게 알림

#### 분산 코디네이터

클러스트의 상태정보(누가 primary인지, 각 복제본들의 버젼은 어떤지)등을 관리하는 key-value 저장소

#### 동작 원리

##### 평상시 상태(TTL 기반 리더 유지)

클러스터가 처음 켜지면 코디네이터에서 키를 선점하는 노드가 primary 노드가 됨 | 이 리더 키에는 TTL이 걸려있음

primary노드의 patroni는 자신의 생존 여부를 주기적으로 코디네이터에게 알림 | 이를 통해 TTL 연장

##### 장애 발생 시 페일오버 메커니즘

TTL 만료 -> primary 사망으로 판정

코디네이터가 저장하고있던 키-값 저장소에서 각 replica버전 확인 후 최신 버전의 노드를 primary로 promotion

#### 코디네이터가 SPOF아닌가?

기본 3대를 띄움(이것 마저도 합의 알고리즘으로 사용)

### 앞단 라우팅(PgBouncer)

PostgreSQL은 커넥션 풀 관리를 위해 PgBouncer를 사용

원래 pgBouncer는 커넥션 풀 압축 목적으로 사용함

페일 오버의 맥락에서는 우선 라우터의 역할을 함 -> 즉, 페일 오버가 돌아서 primary DB IP가 바뀌어도 어플리케이션 단에서는 DB url을 안바꿔도 됨

무중단 쿼리 버퍼링

페일 오버 시 들어오는 쿼리들을 실패로 처리하는 것이 아닌, pgBouncer 메모리 큐에 버퍼해놓음
