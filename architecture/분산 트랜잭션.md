# 분산 트랜잭션

하나의 로컬 트랜잭션(단일 DB 커밋) 경계를 넘어서, 여러 작업 혹은 여러 서비스의 정합성을 맞춰야 하는 문제

[[이벤트 리스너와 트랜잭션]]에서 봤듯이, `@TransactionalEventListener(AFTER_COMMIT)`으로 후속 작업을 커밋 이후로 미뤄도, 그 후속 작업(알림 발송, 재고 차감, 결제 요청, 외부 API 호출 등)은 이미 원래 트랜잭션 바깥에서 실행된다.

-> 즉, 후속 작업이 실패해도 원래 트랜잭션은 롤백되지 않는다.

그래서 "DB는 커밋됐는데 후속 작업은 실패한" 불일치를 어떻게 다룰지가 문제가 되고, 이게 분산 트랜잭션의 출발점이다.

## 왜 그냥 하나의 트랜잭션으로 묶지 못하나?

- 외부 시스템(메일 서버, 결제사, 다른 마이크로서비스)은 우리 DB 트랜잭션에 참여하지 못한다
- 서비스가 나뉘면 DB도 나뉘기 때문에, 하나의 커밋으로 전부를 묶을 수 없다
- 외부 API 호출은 롤백이라는 개념 자체가 없다(이미 보낸 요청을 되돌릴 수 없음)

## 접근 방법

### 2PC (Two-Phase Commit)

- 여러 리소스를 하나의 전역 트랜잭션으로 묶어서 커밋/롤백을 조율하는 방식
- 준비(prepare) 단계와 커밋(commit) 단계로 나뉨
- 강한 정합성을 주지만, 조율자(coordinator) 장애에 취약하고 성능/확장성 부담이 커서 MSA에서는 잘 쓰지 않음

### 보상 트랜잭션 (Saga)

- 각 작업을 로컬 트랜잭션으로 처리하고, 중간에 실패하면 앞서 성공한 작업을 되돌리는 보상 작업을 실행
- ex> 결제 성공 후 재고 차감 실패 -> 결제 취소(환불)라는 보상 작업 수행
- [[이벤트 리스너와 트랜잭션]]의 AFTER_ROLLBACK 사용 예시(실패 알림, 임시 리소스 정리, 보상 트랜잭션)와 연결되는 개념

### Outbox 패턴

- 후속 작업 이벤트를 "곧바로 외부로 보내는" 대신, 원래 트랜잭션 안에서 outbox 테이블에 이벤트를 함께 저장
- DB 커밋과 이벤트 기록이 하나의 로컬 트랜잭션으로 원자적으로 묶임
- 이후 별도 프로세스가 outbox를 읽어 메시지 큐로 발행 -> "DB는 커밋됐는데 이벤트는 유실된" 상황을 막음

### 메시지 큐 + 재시도

- 후속 작업을 메시지 큐에 넣어 비동기로 처리하고, 실패 시 재시도
- 최소 한 번 이상 전달을 보장하되, 중복 실행에 대비해 멱등성(idempotency)을 확보해야 함

## 정리

분산 트랜잭션은 "강한 정합성(2PC)"보다는 대개 "최종적 일관성(eventual consistency)"을 받아들이고, 실패를 감지 -> 재시도/보상으로 수렴시키는 방향으로 설계한다.

- 후속 작업을 트랜잭션 커밋 이후로 분리 -> [[이벤트 리스너와 트랜잭션]]
- 이벤트 유실 방지 -> Outbox
- 실패 복구 -> 재시도 / 보상 트랜잭션(Saga)
